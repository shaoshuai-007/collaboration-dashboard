## 一、需求阶段方法论

### 1.1 需求调研方法

#### 1.1.1 访谈法

| 维度 | 说明 |
|------|------|
| **定义** | 通过与客户面对面交流，获取需求信息 |
| **步骤** | 1. 准备访谈提纲<br>2. 预约访谈对象<br>3. 进行访谈<br>4. 整理访谈记录 |
| **技巧** | 开放式问题、倾听、追问、确认 |
| **适用场景** | 高层访谈、关键用户访谈 |

#### 1.1.2 问卷法

| 维度 | 说明 |
|------|------|
| **定义** | 通过问卷收集大量用户的需求信息 |
| **步骤** | 1. 设计问卷<br>2. 发放问卷<br>3. 收集问卷<br>4. 分析问卷 |
| **技巧** | 问题简洁、选项明确、逻辑清晰 |
| **适用场景** | 大量用户调研、满意度调查 |

#### 1.1.3 观察法

| 维度 | 说明 |
|------|------|
| **定义** | 通过现场观察用户的实际操作，发现需求 |
| **步骤** | 1. 确定观察对象<br>2. 进行现场观察<br>3. 记录观察结果<br>4. 分析观察发现 |
| **技巧** | 不干扰、多角度、记录详细 |
| **适用场景** | 业务流程观察、用户行为观察 |

---

### 1.2 需求分析方法

#### 1.2.1 用户故事法

| 维度 | 说明 |
|------|------|
| **定义** | 用用户的语言描述需求 |
| **格式** | 作为[角色]，我想要[功能]，以便[价值] |
| **示例** | 作为客户，我想要智能问答，以便快速获取业务咨询 |

**用户故事模板**：

| 编号 | 角色 | 功能 | 价值 |
|------|------|------|------|
| US-001 | 客户 | 智能问答 | 快速获取业务咨询 |
| US-002 | 客户 | 智能查询 | 快速查询业务数据 |
| US-003 | 客户 | 智能办理 | 快速办理业务 |

---

#### 1.2.2 用例分析法

| 维度 | 说明 |
|------|------|
| **定义** | 描述系统与用户的交互过程 |
| **要素** | 参与者、用例、前置条件、后置条件、流程 |
| **格式** | 用例图 + 用例描述 |

**用例描述模板**：

| 用例名称 | [用例名称] |
|---------|-----------|
| 参与者 | [参与者] |
| 前置条件 | [前置条件] |
| 后置条件 | [后置条件] |
| 主流程 | 1. [步骤1]<br>2. [步骤2]<br>3. [步骤3] |
| 备选流程 | 1. [备选步骤1]<br>2. [备选步骤2] |

---

#### 1.2.3 痛点分析法

| 维度 | 说明 |
|------|------|
| **定义** | 识别和分析用户的痛点 |
| **步骤** | 1. 痛点识别<br>2. 痛点分析<br>3. 痛点优先级<br>4. 痛点解决 |

**痛点分析模板**：

| 痛点编号 | 痛点描述 | 痛点表现 | 影响范围 | 影响程度 | 解决方案 |
|---------|---------|---------|---------|---------|---------|
| P1 | 咨询效率低 | 响应时间长 | 所有咨询业务 | 高 | 智能问答系统 |
| P2 | 查询效率低 | 操作复杂 | 所有查询业务 | 高 | 智能查询系统 |

### 1.3 需求优先级方法

#### 1.3.1 MoSCoW方法

## 二、方案阶段方法论

### 2.1 头脑风暴方法

#### 2.1.1 SCAMPER法

| 维度 | 说明 | 示例 |
|------|------|------|
| **S（Substitute）** | 替代 | 用AI替代人工问答 |
| **C（Combine）** | 合并 | 合并问答和查询功能 |
| **A（Adjust）** | 调整 | 调整业务流程 |
| **M（Modify）** | 修改 | 修改系统架构 |
| **P（Put to other uses）** | 改变用途 | 知识库用于诊断 |
| **E（Eliminate）** | 消除 | 消除人工操作 |
| **R（Reverse）** | 反转 | 主动推荐替代被动查询 |

---

#### 2.1.2 六顶思考帽

|------|----------|----------|

### 2.2 方案设计方法

#### 2.2.1 SMART原则

| 维度 | 说明 | 示例 |
|------|------|------|
| **S（Specific）** | 具体的 | 建设智能问答系统 |
| **M（Measurable）** | 可衡量的 | 响应时间≤10秒 |
| **A（Achievable）** | 可达成的 | 技术成熟、资源充足 |
| **R（Relevant）** | 相关的 | 解决咨询效率低问题 |
| **T（Time-bound）** | 有时限的 | 2026年6月上线 |

---

#### 2.2.2 方案评估方法

| 维度 | 评分标准 |
|------|----------|
| **影响度** | 5分：影响所有用户<br>4分：影响大部分用户<br>3分：影响部分用户<br>2分：影响少量用户<br>1分：影响个别用户 |
| **可行性** | 5分：技术成熟，资源充足<br>4分：技术较成熟，资源较充足<br>3分：技术一般，资源一般<br>2分：技术较新，资源较紧张<br>1分：技术不成熟，资源不足 |
| **综合得分** | 综合得分 = 影响度 × 0.6 + 可行性 × 0.4 |

---

### 2.3 方案呈现方法

#### 2.3.1 金字塔原理

| 维度 | 说明 |
|------|------|
| **核心观点** | 最上层：核心结论 |
| **支撑论据** | 中间层：支撑论据 |
| **详细证据** | 最下层：详细证据 |

**示例**：

```
核心观点：建设智能营业厅系统，提升效率、优化体验、降低成本

支撑论据：
1. 智能问答系统：提升咨询效率50%
2. 智能查询系统：提升查询效率60%
3. 智能办理系统：提升办理效率70%
4. 智能诊断系统：提升诊断效率60%

详细证据：
- 响应时间：5分钟→10秒
- 查询时间：10分钟→30秒
- 办理时间：30分钟→5分钟
- 诊断时间：20分钟→3分钟
```

## 三、项目阶段方法论

### 3.1 项目分解方法

#### 3.1.1 WBS分解法

| 维度 | 说明 |
|------|------|
| **定义** | 将项目分解为可管理的工作包 |
| **原则** | 100%原则、可交付成果导向、层次清晰 |
| **步骤** | 1. 识别项目阶段<br>2. 分解工作包<br>3. 分配责任 |

**WBS模板**：

```
项目
├─ 阶段1：基础建设
│  ├─ 工作包1.1：知识库建设
│  ├─ 工作包1.2：图谱构建
│  └─ 工作包1.3：数据准备
├─ 阶段2：核心开发
│  ├─ 工作包2.1：问答引擎开发
│  ├─ 工作包2.2：查询引擎开发
│  └─ 工作包2.3：流程引擎开发
└─ 阶段3：集成测试
   ├─ 工作包3.1：前端集成
   ├─ 工作包3.2：系统集成
   └─ 工作包3.3：功能测试
```

---

### 3.2 责任分配方法

#### 3.2.1 RACI矩阵

| 维度 | 说明 |
|------|------|
| **R（Responsible）** | 执行者：负责具体执行 |
| **A（Accountable）** | 负责人：对结果负责 |
| **C（Consulted）** | 咨询者：提供咨询建议 |
| **I（Informed）** | 知情者：了解进展情况 |

**RACI规则**：
- 每个任务必须有且只有一个A
- 每个任务至少有一个R
- C和I可以有多个
- A和R可以是同一个人

---

### 3.3 项目控制方法

#### 3.3.1 甘特图

| 维度 | 说明 |
|------|------|
| **定义** | 可视化项目进度的工具 |
| **要素** | 任务、开始时间、结束时间、负责人、进度 |

---

#### 3.3.2 风险管理方法

| 步骤 | 说明 |
|------|------|
| **风险识别** | 识别可能的风险 |
| **风险评估** | 评估风险概率和影响 |
| **风险应对** | 制定应对措施 |
| **风险监控** | 持续监控风险 |

**风险评估矩阵**：

| 风险等级 | 高影响 | 中影响 | 低影响 |
|---------|--------|--------|--------|
| **高概率** | 高风险 | 高风险 | 中风险 |
| **中概率** | 高风险 | 中风险 | 低风险 |
| **低概率** | 中风险 | 低风险 | 低风险 |
