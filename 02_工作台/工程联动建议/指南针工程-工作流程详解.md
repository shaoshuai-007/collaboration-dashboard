## 二、各步骤详解

### 步骤1：需求采集（采薇）

**目标**：将模糊需求转化为清晰的需求文档

**输入**：
- 口头描述
- 邮件内容
- 会议纪要
- 简要想法

**输出**：需求文档.docx

**内容框架**：
```
一、需求背景
  1.1 项目背景
  1.2 业务现状
  1.3 问题分析

二、需求描述
  2.1 功能需求
  2.2 非功能需求
  2.3 业务场景

三、需求目标
  3.1 业务目标
  3.2 技术目标
  3.3 量化指标

四、约束条件
  4.1 时间约束
  4.2 成本约束
  4.3 资源约束

五、验收标准
  5.1 功能验收标准
  5.2 性能验收标准
  5.3 业务验收标准
```

**方法论**：
- 用户故事（User Story）
- 用例分析（Use Case）
- 需求分级（MoSCoW）

**审核标准**：
- 完整性：需求是否完整
- 正确性：需求是否正确
- 可行性：需求是否可实现
- 优先级：需求是否排序

---

### 步骤2：需求分析（织锦）

**目标**：将需求文档转化为可视化思维导图

**输入**：
- 需求文档.docx
- 需求描述文本

**输出**：思维导图.html

**分析维度**：
```
核心主题
├── 需求理解
│   ├── 核心需求
│   ├── 次要需求
│   └── 隐性需求
├── 痛点分析
│   ├── 效率痛点
│   ├── 体验痛点
│   └── 成本痛点
├── 解决方案
│   ├── 方案1
│   ├── 方案2
│   └── 方案3
└── 优先级评估
    ├── 影响度
    ├── 可行性
    └── 综合评估
```

**方法论**：
- 痛点分析法
- SCAMPER法
- 优先级矩阵

**审核标准**：
- 结构清晰：层次分明
- 逻辑严谨：因果关系明确
- 重点突出：核心痛点识别准确

---

### 步骤3：方案设计（筑台）

**目标**：将思维导图转化为具体的方案举措

**输入**：
- 思维导图.html
- 需求文档.docx

**输出**：方案举措.xlsx

**内容框架**：
```
| 序号 | 方案标题 | 建设目标 | 具体举措 | 预期效果 | PPT页码 |
|------|----------|----------|----------|----------|---------|
| 1    | 智能问答 | 目标描述 | 举措列表 | 效果描述 | 4-5     |
| 2    | 智能配案 | 目标描述 | 举措列表 | 效果描述 | 6-7     |
```

**方法论**：
- SMART原则（目标量化）
- 方案对比法
- 成本收益分析

**审核标准**：
- 目标量化：是否有具体指标
- 举措具体：是否可执行
- 效果可衡量：是否有预期效果

---

### 步骤4：PPT设计（呈彩）

**目标**：将方案举措转化为专业的方案PPT

**输入**：
- 方案举措.xlsx
- 思维导图.html

**输出**：方案PPT.pptx

**设计规范**：

**配色方案**（三原色定理 60-30-10）：
```
主色（60%）：电信红 #C93832
- 标题、重点、核心元素、封面背景

辅色（30%）：深蓝 #006EBD
- 次要标题、图表、系统元素

中性色（10%）：灰色 #595959
- 正文文字、背景、分隔线
```

**布局规范**：
```
左右布局：
┌──────────────┬──────────────┐
│              │              │
│   文字内容   │   图表/图片  │
│              │              │
└──────────────┴──────────────┘

上下布局：
┌──────────────────────────────┐
│          标题区域             │
├──────────────────────────────┤
│                              │
│         内容区域             │
│                              │
└──────────────────────────────┘
```

**故事线设计**：
1. 封面：项目名称 + 副标题
2. 背景：项目背景 + 现状分析
3. 痛点：问题识别 + 影响分析
4. 方案：解决方案 + 核心价值
5. 落地：实施计划 + 预期效果
6. 结语：总结 + 下一步行动

**审核标准**：
- 配色规范：严格遵守三原色定理
- 布局合理：左右/上下布局清晰
- 故事完整：每页有明确主线
- 图文呼应：图片与内容匹配

---

### 步骤5：详细设计（工尺）

**目标**：将评审通过的PPT转化为详细的系统设计文档

**输入**：
- 方案PPT.pptx（评审通过）

**输出**：详细设计.docx

**内容框架**：
```
一、系统架构设计
  1.1 总体架构
  1.2 技术架构
  1.3 部署架构
  1.4 六层架构表

二、接口设计
  2.1 接口规范
  2.2 接口列表
  2.3 接口详情

三、数据库设计
  3.1 概念模型
  3.2 逻辑模型
  3.3 物理模型

四、UI设计
  4.1 界面原型
  4.2 交互设计
  4.3 视觉规范

五、安全设计
  5.1 安全架构
  5.2 安全措施
  5.3 安全审计
```

**方法论**：
- 架构设计方法（六层架构）
- 接口设计方法（RESTful API）
- 数据库设计方法（E-R模型）

**审核标准**：
- 架构完整：六层架构齐全
- 接口规范：RESTful规范
- 数据库合理：范式设计
- 安全可控：安全措施到位

---

### 步骤6：项目管控（玉衡）

**目标**：为项目实施提供全面的管控计划

**输入**：
- 详细设计.docx

**输出**：项目管控.xlsx（5个Sheet）

**内容框架**：

**Sheet1：甘特图**
```
| 任务名称 | 开始时间 | 结束时间 | 负责人 | 进度 | 状态 |
|----------|----------|----------|--------|------|------|
| 需求确认 | 2026-03-13 | 2026-03-20 | 张三 | 100% | 完成 |
| 架构设计 | 2026-03-20 | 2026-03-27 | 李四 | 80% | 进行中 |
```

**Sheet2：RACI矩阵**
```
| 任务 | 项目经理 | 架构师 | 开发 | 测试 | 运维 |
|------|----------|--------|------|------|------|
| 需求确认 | A | R | C | I | I |
| 架构设计 | A | R | C | I | C |

说明：R=执行者, A=负责人, C=咨询者, I=知情者
```

**Sheet3：风险清单**
```
| 风险项 | 风险等级 | 影响 | 应对措施 | 责任人 |
|--------|----------|------|----------|--------|
| 需求变更 | 高 | 项目延期 | 需求冻结机制 | 项目经理 |
```

**Sheet4：资源表**
```
| 资源类型 | 资源名称 | 数量 | 开始时间 | 结束时间 |
|----------|----------|------|----------|----------|
| 人力资源 | 项目经理 | 1 | 2026-03-13 | 2026-06-13 |
```

**Sheet5：沟通计划**
```
| 沟通事项 | 频率 | 对象 | 方式 | 负责人 |
|----------|------|------|------|--------|
| 项目周会 | 每周一 | 项目组 | 会议 | 项目经理 |
```

**方法论**：
- WBS分解（工作分解结构）
- RACI矩阵（责任分配矩阵）
- 甘特图（进度管理）
- 风险管理（风险识别与应对）

**审核标准**：
- 任务完整：WBS分解到位
- 责任明确：RACI清晰
- 进度可控：甘特图合理
- 风险识别：风险清单完整

## 三、人工评审点

### 评审位置

**步骤4完成后**（PPT生成后）

### 评审内容

| 评审项 | 评审标准 | 审核等级 |
|--------|----------|----------|
| 需求完整性 | 需求是否完整、正确 | A/B/C/D |
| 方案合理性 | 方案是否可行、有价值 | A/B/C/D |
| PPT质量 | 配色、布局、故事线 | A/B/C/D |
| 综合评估 | 整体质量评估 | A/B/C/D |

### 审核等级

| 等级 | 分数 | 处理方式 |
|------|------|----------|
| A级 | 90-100分 | 直接通过，继续下一步 |
| B级 | 80-89分 | 小幅优化后继续 |
| C级 | 70-79分 | 重新审核后决定 |
| D级 | <70分 | 返回修改，重新生成 |

### 评审流程

```
步骤4完成 → 用户评审 → 决策
                         │
                         ├── A级 → 继续
                         ├── B级 → 优化后继续
                         ├── C级 → 重新审核
                         └── D级 → 返回修改
```
