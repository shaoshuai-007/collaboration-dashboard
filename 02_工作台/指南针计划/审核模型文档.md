## 一、审核体系总览

### 1.1 双引擎审核架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                         审核体系架构                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   执行引擎输出                                                       │
│        │                                                            │
│        ↓                                                            │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                 流程审核引擎（Auditor）                       │   │
│   │                                                             │   │
│   │   检查：步骤完整性 + 方法论正确性 + 输出物质量 + 时间合规      │   │
│   │                                                             │   │
│   │   判定：🟢通过 / 🟡修正 / 🔴退回                             │   │
│   └─────────────────────────────────────────────────────────────┘   │
│        │                                                            │
│        ↓ 通过                                                       │
│                                                                     │
│   进入下一步（循环审核）                                              │
│        │                                                            │
│        ↓ 全部完成                                                   │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                 价值审核引擎（Validator）                     │   │
│   │                                                             │   │
│   │   检查：问题解决 + 目标达成 + 方案可行 + 知识沉淀             │   │
│   │                                                             │   │
│   │   判定：S级 / A级 / B级 / C级                                │   │
│   └─────────────────────────────────────────────────────────────┘   │
│        │                                                            │
│        ↓                                                            │
│                                                                     │
│   入库保存 / 优化打磨                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 审核原则

| 原则 | 说明 |
|------|------|
| **独立审核** | 审核引擎独立于执行引擎，避免自我审核 |
| **标准驱动** | 所有审核基于《方案输出标准体系》 |
| **即时反馈** | 每步完成后立即审核，快速迭代 |
| **价值导向** | 最终审核关注价值产出，而非形式合规 |

## 二、流程审核模型（Auditor）

### 2.1 模型定位

| 属性 | 内容 |
|------|------|
| 模型名称 | 流程审核模型 |
| 模型代号 | Auditor |
| 核心目标 | 保障流程规范、质量可靠 |
| 触发时机 | 每个步骤完成后 |

### 2.2 审核维度

#### 维度1：步骤完整性（权重30%）

**检查内容**：关键步骤是否完成

**检查标准**：

| 步骤 | 分类 | 必需项 | 检查标准 |
|:----:|:----:|--------|----------|
| 0 | 关键 | 5W2H分析 | 7个维度全部覆盖 |
| 0 | 关键 | SWOT分析 | 4个象限均有内容 |
| 0 | 关键 | 根因分析 | 至少追问5层 |
| 1 | 关键 | SMART目标 | 5个要素全部具备 |
| 1 | 关键 | OKR分解 | 目标+3-5个KR |
| 1 | 关键 | WBS分解 | 至少3层分解 |
| 2 | 关键 | 六顶思考帽 | 6种颜色全部使用 |
| 2 | 关键 | 方案数量 | 至少3个方案方向 |
| 2.5 | 重要 | 干系人地图 | 四象限均有干系人 |
| 2.5 | 重要 | 用户故事 | 至少5个用户故事 |
| 3 | 关键 | WBS分解 | 至少4层分解 |
| 3 | 关键 | RACI矩阵 | 每个子任务有R和A |
| 3.5 | 重要 | 可行性结论 | 明确可行/不可行 |
| 4 | 关键 | PPT结构 | 标准章节完整 |
| 5 | 关键 | 知识卡片 | 标准格式完整 |

**评分标准**：

| 完成度 | 得分 |
|:------:|:----:|
| 100%必需项完成 | 30分 |
| 80%必需项完成 | 24分 |
| 60%必需项完成 | 18分 |
| <60%必需项完成 | 0分 |

---

#### 维度2：方法论正确性（权重30%）

**检查内容**：必需方法论是否正确使用

**检查标准**：

| 步骤 | 必需方法论 | 正确使用标准 |
|:----:|------------|--------------|
| 0 | 5W2H | 每个维度有明确答案 |
| 0 | SWOT | 四象限分类正确 |
| 0 | 5Why | 追问逻辑连贯 |
| 0 | 鱼骨图 | 至少覆盖4个维度 |
| 1 | SMART | 目标符合5要素 |
| 1 | OKR | KR可量化可衡量 |
| 1 | WBS | 分解逻辑清晰 |
| 1 | RACI | 角色分配合理 |
| 2 | 六顶思考帽 | 每种颜色思考正确 |
| 2 | SCAMPER | 至少使用5种方法 |
| 2.5 | KANO | 三类需求分类正确 |
| 2.5 | MoSCoW | 优先级排序合理 |
| 3 | WBS | 分解层次清晰 |
| 3 | RACI | 每个任务有唯一R |
| 3.5 | 成本收益分析 | ROI计算合理 |
| 4 | 金字塔原理 | 结论先行、层层支撑 |
| 4 | SCQA | 情境-冲突-问题-答案完整 |
| 5 | AAR | 六问全部回答 |

**评分标准**：

| 正确率 | 得分 |
|:------:|:----:|
| 100%正确 | 30分 |
| 80%正确 | 24分 |
| 60%正确 | 18分 |
| <60%正确 | 0分 |

---

#### 维度3：输出物质量（权重30%）

**检查内容**：必需内容是否包含

**检查标准**：

| 步骤 | 输出物 | 必需内容 |
|:----:|--------|----------|
| 0 | 问题分析文档 | 问题陈述、5W2H、SWOT、根因、鱼骨图、问题定义 |
| 1 | 管控文档 | SMART目标、OKR、WBS、RACI、时间规划 |
| 2 | 思维导图 | 六顶思考帽分析、SCAMPER分析、方案分类、可视化结构 |
| 2.5 | 需求规格说明书 | 干系人地图、用户故事、KANO分类、MoSCoW排序 |
| 3 | 详细举措表 | WBS分解、RACI矩阵、时间估算、优先级、资源规划 |
| 3.5 | 可行性评估报告 | 技术评估、经济评估、操作评估、风险评估、结论 |
| 4 | 方案PPT | 封面、目录、背景、问题、方案、举措、收益、附录 |
| 5 | 知识卡片 | 背景、过程、结果、经验、适用边界 |

**评分标准**：

| 质量等级 | 得分 |
|:--------:|:----:|
| 内容完整、质量优秀 | 30分 |
| 内容完整、质量良好 | 24分 |
| 内容基本完整 | 18分 |
| 内容缺失 | 0分 |

---

#### 维度4：时间节点合规（权重10%）

**检查内容**：是否按时交付

**检查标准**：

| 情况 | 得分 |
|------|:----:|
| 按时交付 | 10分 |
| 延期但有合理说明 | 7分 |
| 延期无说明 | 3分 |
| 严重延期 | 0分 |

---

### 2.3 判定标准

| 总分 | 结果 | 处理方式 |
|:----:|:----:|----------|
| ≥90分 | 🟢 通过 | 直接进入下一步 |
| 70-89分 | 🟡 需修正 | 修正后通过，无需重新审核 |
| 60-69分 | 🟠 需重审 | 修正后重新提交审核 |
| <60分 | 🔴 退回 | 返回重做该步骤 |

### 2.4 审核流程

```
步骤完成 → 提交审核 → 读取标准 → 检查4个维度 → 计算总分 → 判定结果
                                                      ↓
                                            ┌────────┼────────┐
                                            ↓        ↓        ↓
                                          通过     修正     退回
                                            ↓        ↓        ↓
                                        下一步   修正后    重做
                                                  继续
```

## 三、价值审核模型（Validator）

### 3.1 模型定位

| 属性 | 内容 |
|------|------|
| 模型名称 | 价值审核模型 |
| 模型代号 | Validator |
| 核心目标 | 评估产出价值、指导入库决策 |
| 触发时机 | 步骤5完成后 / 最终输出时 |

### 3.2 审核维度

#### 维度1：问题解决度（权重30%）

**评估内容**：问题是否得到解决

**评估标准**：

| 检查项 | 标准问题 | 评分标准 |
|--------|----------|----------|
| 问题定义 | 问题是否清晰定义？ | 清晰10分/模糊5分/缺失0分 |
| 解决方案 | 是否有明确的解决方案？ | 明确10分/笼统5分/缺失0分 |
| 效果验证 | 是否有验证效果的方法？ | 有方法10分/无方法0分 |

**综合评分**：

| 总分 | 等级 |
|:----:|:----:|
| 25-30分 | 优秀 |
| 20-24分 | 良好 |
| 15-19分 | 合格 |
| <15分 | 不合格 |

---

#### 维度2：目标达成度（权重25%）

**评估内容**：目标是否达成

**评估标准**：

| 检查项 | 评估方法 | 评分标准 |
|--------|----------|----------|
| OKR目标 | 目标vs实际 | 达成率×10分 |
| 关键结果 | KR完成情况 | 完成率×10分 |
| 里程碑 | 节点完成情况 | 完成率×5分 |

**计算公式**：

```
目标达成度得分 = (OKR达成率 × 10 + KR完成率 × 10 + 里程碑完成率 × 5) / 25 × 权重
```

---

#### 维度3：方案可行性（权重25%）

**评估内容**：方案是否可落地

**评估标准**：

| 检查项 | 评估内容 | 评分标准 |
|--------|----------|----------|
| 技术可行性 | 技术方案是否成熟？ | 成熟10分/待验证5分/不可行0分 |
| 经济可行性 | ROI是否合理？ | ROI>1得10分/ROI=1得5分/ROI<1得0分 |
| 操作可行性 | 资源是否可获取？ | 可获取10分/部分可获取5分/不可获取0分 |

**综合评分**：

| 总分 | 等级 |
|:----:|:----:|
| 22-30分 | 高可行性 |
| 15-21分 | 中等可行性 |
| 8-14分 | 低可行性 |
| <8分 | 不可行 |

---

#### 维度4：知识沉淀度（权重20%）

**评估内容**：是否形成有效沉淀

**评估标准**：

| 检查项 | 评估内容 | 评分标准 |
|--------|----------|----------|
| 知识卡片 | 是否形成标准卡片？ | 完整10分/部分5分/无0分 |
| 可复用性 | 是否可复用？ | 高复用5分/中复用3分/低复用1分 |
| 入库保存 | 是否已入库？ | 已入库5分/未入库0分 |

**综合评分**：

| 总分 | 等级 |
|:----:|:----:|
| 16-20分 | 高沉淀度 |
| 10-15分 | 中沉淀度 |
| 5-9分 | 低沉淀度 |
| <5分 | 无沉淀 |

---

### 3.3 价值评级标准

| 总分 | 评级 | 说明 | 处理 |
|:----:|:----:|------|------|
| ≥90分 | **S级** | 价值连城，可直接推广 | 立即入库，优先推广 |
| 80-89分 | **A级** | 高价值，建议采纳 | 入库保存，推广应用 |
| 70-79分 | **B级** | 有价值，优化后采纳 | 入库保存，优化推广 |
| 60-69分 | **C级** | 有参考价值 | 入库保存，待优化 |
| <60分 | **D级** | 需重新打磨 | 返回优化，重新审核 |

### 3.4 评估流程

```
全部完成 → 提交价值审核 → 四维评估 → 计算总分 → 价值评级
                                              ↓
                                    ┌─────────┼─────────┐
                                    ↓         ↓         ↓
                                  S/A级      B/C级      D级
                                    ↓         ↓         ↓
                                立即入库   入库优化   返回重做
```

## 五、审核异常处理

### 5.1 审核争议

| 情况 | 处理方式 |
|------|----------|
| 对审核结果有异议 | 提供申诉材料，人工复核 |
| 标准理解有偏差 | 查阅标准文档，统一解释 |
| 特殊情况 | 上报少帅，特殊处理 |

### 5.2 审核超时

| 情况 | 处理方式 |
|------|----------|
| 审核引擎故障 | 人工审核，事后补充 |
| 标准文档缺失 | 参考历史案例，补充标准 |
