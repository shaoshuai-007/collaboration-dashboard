**客户单位**：中国电信集团市场部

## 一、项目背景

### 1.1 业务背景

为贯彻落实集团"云改数转智惠"战略，实现全集团AI应用Token消耗数据的集中汇聚和统一管理，市场部提出建设Token采集系统。

### 1.2 现状分析

| 维度 | 现状 | 问题 |
|------|------|------|
| 数据分散 | 各省公司、专业公司AI应用数据独立存储 | 无法统一汇总分析 |
| 标准缺失 | Token统计口径不统一 | 数据不可比 |
| 管控困难 | 缺乏全集团视角的成本管控 | AI投入产出不可评估 |

### 1.3 建设目标

实现"三个统一"：
- **统一采集**：全集团Token数据集中汇聚
- **统一标准**：规范数据格式和统计口径
- **统一管控**：实现AI成本可视化管控

│  └─────────────┘  └─────────────┘  └─────────────┘        │
│         ↑                                      ↓            │
│  ┌─────────────┐                      ┌─────────────┐      │
│  │ SFTP服务    │                      │ 大数据平台  │      │
│  │ (10.141.208.176:12222)             └─────────────┘      │

         ↑
         │ SFTP上传
         │

### 3.2 核心流程

**步骤1：数据准备（各省公司）**
1. 从大模型调用日志中抽取Token消耗明细
2. 按规范生成数据文件（DAT.gz格式）
3. 生成VAL校验文件（记录数、文件大小、MD5）
4. 生成CHECK文件（文件清单）

**步骤2：数据上传（各省公司）**
1. 通过SFTP上传至集团接口机
2. 上传顺序：数据文件→VAL文件→CHECK文件
3. 上传完成后去掉.TMP后缀

**步骤3：数据稽核（集团大数据中心）**
1. CHECK文件完整性校验
2. VAL文件记录数校验
3. VAL文件MD5校验
4. 生成回执文件（RPT/ERR）

**步骤4：数据入库（集团大数据中心）**
1. 解析数据文件
2. 数据清洗转换
3. 写入大数据平台

### 3.3 文件规范

**文件命名格式**：
```
编码_数据来源_数据来源系统_接口编码_文件处理时间_文件数据时间_文件上传周期_文件上传批次_文件序列号.后缀

示例：
10800_JWJCB_下属系统名称_AIP_MDL_TOKEN_INFO_20260117_20260116_D_00_0001.DAT.gz
```

**字段说明**：

| 序号 | 字段 | 说明 |
|:----:|------|------|
| 1 | 编码 | 固定"10800" |
| 2 | 数据来源 | 单位编码（见主数据） |
| 3 | 数据来源系统 | 系统名称（4位英文） |
| 4 | 接口编码 | AIP_MDL_TOKEN_INFO |
| 5 | 文件处理时间 | YYYYMMDD |
| 6 | 文件数据时间 | YYYYMMDD（实际Token消耗日期） |
| 7 | 文件上传周期 | D（日） |
| 8 | 文件上传批次 | 00（首次）、01（第一次重传） |
| 9 | 文件序列号 | 0001开始 |
| 10 | 后缀 | DAT.gz/VAL/CHECK/RPT/ERR |
