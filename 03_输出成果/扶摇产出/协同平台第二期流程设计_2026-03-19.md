### 2.2 三阶段流程详解

#### 阶段一：多Agent讨论（V15.1已有能力）

| 环节 | 执行者 | 输入 | 输出 |
|:----:|:------:|------|------|
| 任务解析 | 🌀 扶摇 | 用户输入 | 任务类型、Agent调度表 |
| 角色发言 | 各Agent | 任务背景 | 专业观点、建议、质疑 |
| 质疑响应 | 被质疑Agent | 质疑内容 | 解释、修正、补充 |
| 共识计算 | 系统 | 发言记录 | 共识度分数 |

#### 阶段二：讨论结束判断（V15.2新增能力）

| 判断条件 | 阈值 | 说明 |
|:--------:|:----:|------|
| 共识度达标 | ≥70% | 团队达成基本共识 |
| 无未响应质疑 | 0条 | 所有质疑均有回应 |
| 轮次达到上限 | 15轮 | 防止无限讨论 |
| 用户手动结束 | 用户确认 | 用户主动结束讨论 |

**结束判断逻辑**：
```python
def is_discussion_complete():
    # 任意条件满足即结束
    if consensus_score >= 70:
        return True, "共识度达标"
    if pending_challenges == 0 and turn >= 5:
        return True, "无待响应质疑"
    if turn >= 15:
        return True, "达到轮次上限"
    if user_ended:
        return True, "用户手动结束"
    return False, "继续讨论"
```

#### 阶段三：产物生成（V15.2核心能力）

| 环节 | 执行者 | 核心动作 |

## 三、触发机制

### 3.1 讨论结束触发

**自动触发条件**（任一满足）：

| 序号 | 条件 | 阈值 | 触发动作 |
|:----:|------|:----:|----------|
| 1 | 共识度达标 | ≥70% | 自动进入产物生成 |
| 2 | 无未响应质疑 | 0条 | 自动进入产物生成 |
| 3 | 轮次上限 | 15轮 | 自动进入产物生成 |

**手动触发条件**：
- 用户输入"结束讨论"、"生成产物"等指令
- 用户点击"结束讨论"按钮

### 3.2 产物生成触发

**触发流程**：
```
讨论结束信号
    ↓
扶摇接收信号
    ↓
分析讨论内容
    ├── 提取关键信息（任务类型、需求要点、技术方案等）
    ├── 匹配产物类型（需求文档、架构设计、方案PPT等）
    └── 确定产物类型
    ↓
调度专属Agent
    ↓
Agent执行生成
    ↓
质量检查
    ├── 通过 → 输出产物
    └── 不通过 → 返修优化
```

### 3.3 智能产物类型识别

**识别方法**：基于任务类型关键词 + 讨论内容语义分析

| 任务类型 | 关键词示例 | 对应产物 | 负责Agent |
|:--------:|-----------|:--------:|:---------:|
| 需求分析 | 需求、功能、用户故事 | 需求规格说明书 | 🌸 采薇 |
| 架构设计 | 架构、技术选型、系统设计 | 架构设计文档 | 🧵 织锦 |
| 方案设计 | 方案、PPT、演示 | 方案PPT | 🎨 呈彩 |
| 详细设计 | 接口、数据库、API | 详细设计文档 | 📐 工尺 |
| 开发实现 | 代码、开发、实现 | 代码/技术文档 | 💻 天工 |
| 数据分析 | 数据、分析、报表 | 数据分析报告 | 📊 知微 |
| 售前方案 | 报价、竞品、话术 | 售前方案/报价单 | 🏗️ 筑台 |
| 项目管理 | 计划、进度、风险 | 项目计划/管控表 | ⚖️ 玉衡 |

**复合产物处理**：
```python
def identify_product_types(discussion_content):
    """识别产物类型"""
    product_types = []
    
    # 1. 基于任务类型匹配
    for task_type in task_types:
        if match_keywords(discussion_content, task_type.keywords):
            product_types.append(task_type.product)
    
    # 2. 基于语义分析补充
    if "架构图" in discussion_content:
        product_types.append("架构图")
    if "数据库" in discussion_content:
        product_types.append("数据库设计")
    
    # 3. 去重并排序
    product_types = list(set(product_types))
    product_types = sort_by_priority(product_types)
    
    return product_types
```

### 4.2 协作模式

**模式一：独立产出**
- 单一Agent独立完成产物
- 适用场景：需求文档、详细设计、代码实现等
- 流程：扶摇调度 → Agent执行 → 南乔质检 → 输出

**模式二：协作产出**
- 主责Agent + 协作Agent共同完成
- 适用场景：架构设计、方案PPT等
- 流程：扶摇调度 → 主责Agent主导 → 协作Agent补充 → 集成优化 → 南乔质检 → 输出

**模式三：链式产出**
- 多Agent按工作流顺序产出
- 适用场景：完整项目交付
- 流程：采薇 → 织锦 → 筑台 → 呈彩 → 工尺 → 天工 → 知微 → 玉衡

## 五、质量把控机制

### 5.1 四维质量检查模型

| 维度 | 检查项 | 达标标准 | 检查方式 |
|:----:|--------|:--------:|:--------:|
| **完整性** | 内容是否完整覆盖任务需求 | 遗漏率<5% | 清单对照 |
| **规范性** | 是否符合模板和格式规范 | 格式正确率100% | 模板匹配 |
| **一致性** | 内容前后是否一致 | 矛盾点=0 | 语义分析 |
| **可落地性** | 方案是否具备可执行性 | 返工率<10% | 专家评审 |

### 5.2 质量检查关卡

**关卡一：产物生成后自检**
- 执行Agent自检产物完整性
- 对照模板检查格式规范
- 记录自检结果

**关卡二：南乔质量把关**
- 四维质量检查
- 评分计算
- 决定通过/返修

**关卡三：团队评审（高风险产物）**
- 高风险产物触发团队评审
- 多Agent交叉评审
- 共识决策

### 5.3 质量评分算法

```python
def calculate_quality_score(product):
    """计算产物质量分数"""
    score = 0
    
    # 完整性（30分）
    completeness = check_completeness(product)
    score += completeness * 30
    
    # 规范性（25分）
    standardization = check_standardization(product)
    score += standardization * 25
    
    # 一致性（20分）
    consistency = check_consistency(product)
    score += consistency * 20
    
    # 可落地性（25分）
    feasibility = check_feasibility(product)
    score += feasibility * 25
    
    return score

def get_quality_grade(score):
    """获取质量等级"""
    if score >= 90:
        return "A级（优秀）"
    elif score >= 80:
        return "B级（良好）"
    elif score >= 70:
        return "C级（合格）"
    else:
        return "D级（需返修）"
```

### 5.4 返修优化机制

**返修触发条件**：
- 质量分数<80分
- 任一维度得分<60分
- 存在关键遗漏或错误

**返修流程**：
```
返修触发
    ↓
南乔分析问题
    ├── 完整性问题 → 补充内容
    ├── 规范性问题 → 调整格式
    ├── 一致性问题 → 修正矛盾
    └── 可落地性问题 → 优化方案
    ↓
返修意见反馈给执行Agent
    ↓
执行Agent优化产物
    ↓
重新质量检查
    ├── 通过 → 输出
    └── 不通过 → 再次返修（最多3次）
```

**返修次数限制**：
- 单个产物最多返修3次
- 3次后仍不达标，南乔接管人工处理

### 5.5 A级产物标准

| 维度 | A级标准 | 具体要求 |
|:----:|:-------:|----------|
| 完整性 | ≥90分 | 内容完整覆盖需求，无关键遗漏 |
| 规范性 | ≥90分 | 严格遵循模板，格式统一规范 |
| 一致性 | ≥90分 | 内容前后一致，无矛盾冲突 |
| 可落地性 | ≥90分 | 方案具体可行，具备执行条件 |
| **总分** | **≥90分** | **A级产物** |

### 6.3 关键算法

**产物类型识别算法**：
```python
def identify_product_types(discussion_state: DiscussionState) -> List[str]:
    """基于讨论内容识别产物类型"""
    
    # 1. 提取讨论关键信息
    discussion_content = get_discussion_content(discussion_state)
    keywords = extract_keywords(discussion_content)
    
    # 2. 任务类型匹配
    task_product_mapping = {
        "需求分析": ["需求规格说明书", "需求追溯矩阵"],
        "架构设计": ["架构设计文档", "系统架构图"],
        "方案设计": ["方案PPT", "方案文档"],
        "详细设计": ["详细设计文档", "接口设计文档", "数据库设计"],
        "开发实现": ["代码实现", "技术文档"],
        "数据分析": ["数据分析报告", "可视化图表"],
        "售前方案": ["售前方案", "报价单"],
        "项目管理": ["项目计划", "风险清单"]
    }
    
    # 3. 匹配产物类型
    product_types = []
    for keyword in keywords:
        for task_type, products in task_product_mapping.items():
            if keyword in task_type:
                product_types.extend(products)
    
    # 4. 去重
    return list(set(product_types))
```

## 七、风险与应对

### 7.1 风险清单

| 风险 | 等级 | 影响 | 应对措施 |
|------|:----:|------|----------|
| 产物类型识别错误 | 中 | 调度错误Agent | 多关键词匹配 + 用户确认 |
| 质量检查误判 | 低 | 返修浪费 | 人工复核机制 |
| Agent执行失败 | 中 | 产物无法生成 | 失败重试 + 备用Agent |
| 讨论无法达成共识 | 低 | 产物质量下降 | 人工介入 + 低共识度标记 |

### 7.2 降级策略

| 场景 | 降级方案 |
|------|----------|
| 产物类型识别失败 | 输出"会议纪要"作为兜底产物 |
| 质量检查异常 | 跳过自动检查，人工评审 |
| Agent执行超时 | 缩短产物范围，输出简化版本 |
| 讨论无限循环 | 强制结束，输出当前讨论总结 |
