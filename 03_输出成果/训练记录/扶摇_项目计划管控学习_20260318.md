## 一、项目协调机制

### 1.1 分层管控架构

从模板中学习到项目管控采用**分层视图**：

| 层级 | 内容 | 更新频度 | 性质 |
|------|------|----------|------|
| 总体视图 | 项目整体管控框架 | - | 导航 |
| 组织架构 | 团队人员与职责 | 有变化则更新 | 基础 |
| 项目周报 | 本周进展+下周计划+问题风险 | 按周 | **必选** |
| 里程碑计划 | 关键里程碑监控 | 按周 | **必选** |
| 总体计划 | 全任务分解与进度 | 按周/日 | **必选** |
| 阶段计划 | 需求/设计/开发/测试/实施 | 根据项目情况 | 可选 |

### 1.2 协调各阶段计划的方法

**实战案例启发**：商机项目将总体计划分解为多个阶段计划：
- 3.3业务组计划（需求方案、UI设计、业务梳理）
- 3.4设计组计划（详细设计、数据模型）
- 3.5商机笔记本开发计划
- 3.6商机中台开发计划
- 3.7商机嗅探开发计划
- 3.8外围接口开发计划
- 3.9测试计划
- 3.10实施计划

**协调原则**：
1. **里程碑驱动**：各阶段计划必须对齐里程碑截止时间
2. **依赖关系显性化**：通过"前置任务"字段明确依赖
3. **责任人明确**：每个任务都有唯一责任人
4. **进度可视化**：统一使用进度百分比+当前进展描述

### 1.3 处理依赖关系的方法

从模板3.2总体计划样例中学到：

```
前置任务 → 详细任务 → 输出物 → 当前进展
```

**依赖处理策略**：
1. 在计划中明确标注前置任务编号
2. 使用里程碑作为关键检查点（"是否里程碑"字段）
3. 按阶段顺序：需求分析→系统设计→软件开发→系统测试→实施部署

## 二、资源调配方法

### 2.1 资源识别维度

从组织架构表学到的资源分类：

| 维度 | 说明 |
|------|------|
| 内外部 | 区分局方资源（安徽电信/湖北电信）和乙方资源（迪科） |
| 角色 | 总体负责人、需求负责人、设计负责人、开发负责人等 |
| 职责描述 | 明确每个角色的具体职责范围 |
| 联系方式 | 便于快速沟通协调 |

### 2.2 识别资源瓶颈

从实战案例中的风险识别学到：

**资源瓶颈信号**：
1. 多个任务依赖同一资源（如"统一门户数据同步"）
2. 外部依赖无法控制进度（如"承包助手接口升级"）
3. 新技术能力缺口（如"大模型不支持图片/语音/视频"）

**案例**：商机项目中识别到：
- "湖北目前只有Qwen-72B的大模型，不支持图片、语音、视频" → 需要IBOC提供API支持
- "生产环境DMZ集群无法访问DCN集群" → 需要网络资源协调

### 2.3 优化资源分配

**方法**：
1. **优先级排序**：高优先级问题优先分配资源
2. **并行推进**：将无依赖的任务分配给不同人员并行执行
3. **关键路径保护**：里程碑任务优先保障资源

**案例**：商机项目通过多工作组并行：
- 业务组：需求方案输出
- 设计组：详细设计输出
- 开发组：各模块并行开发（笔记本/中台/嗅探/外围接口）

## 三、风险预警机制

### 3.1 风险识别框架

从风险参考指引中学到**28项风险识别清单**：

**需求类风险**：
- 需求的稳定性、完整性、清晰性、合理性、可行性
- 需求变化控制机制
- 需求范围风险

**设计类风险**：
- 设计的功能性、难度、接口、性能、易测性
- 硬件约束、可实现性

**开发类风险**：
- 编码及单元测试可行性
- 集成测试环境
- 可维护性、可靠性、安全性

**管理类风险**：
- 开发体系、过程管理、人员管理
- 质量保证、配置管理

### 3.2 风险评估维度

从风险跟踪报告模板学到**五维评估法**：

| 维度 | 取值 |
|------|------|
| 类型 | 进度类、需求类、资源类、成本类、政策类 |
| 发生概率 | 极有可能发生、有可能发生、不太可能发生 |
| 影响程度 | 影响大、影响中、影响小 |
| 发生时机 | 识别日期起2周以内、1个月以上 |
| 级别 | 高、中、低 |

**级别判定规则**：
- 高级别：必须制定预防措施和应急措施
- 低级别：可接受监控

### 3.3 风险应对措施

从实战案例中学到的**双措施机制**：

1. **预防措施**：事前降低发生概率
2. **应急措施**：事后降低影响程度

**案例**：
| 风险 | 预防措施 | 应急措施 |
|------|----------|----------|
| 小程序审核风险 | 提前发起审核申请 | 准备备案材料 |
| 大模型能力不足 | 协调IBOC提供API | 选用云上模型 |
| 异网短信问题 | 判断号段调用三网接口 | 预留备用通道 |

## 四、里程碑监控方法

### 4.1 里程碑设置原则

从模板中学到的规则：

1. **数量要求**：每月至少1个里程碑（含初验、终验）
2. **完整性要求**：项目结项视为里程碑计划制定完整
3. **交付物要求**：里程碑计划与完成必须有明确的交付物
4. **对标要求**：里程碑计划按功能对标一级功能模块制定

### 4.2 偏差判定方法

**延迟判定规则**：
- 当进度≠100%，且计划完成时间≥当天 → 非延迟
- 当进度=100%，且实际完成时间≤计划完成时间 → 非延迟
- 其余情况 → 延迟

**偏差状态判定**：
- 异常：偏差>15%（(当日-计划完成时间)/工时>15%）
- 可控：偏差≤15%
- 正常：无偏差

### 4.3 监控要素

从实战案例中学到的监控字段：

| 字段 | 说明 |
|------|------|
| 里程碑名称 | 清晰描述里程碑目标 |
| 计划交付物 | 明确产出物清单 |
| 完成标准 | 可验证的验收条件 |
| 计划完成时间 | 截止日期 |
| 进度% | 当前完成百分比 |
| 进度说明 | 详细进展描述 |
| 实际完成时间 | 实际达成日期 |
| 实际工时 | 实际投入天数 |
| 实际输出物 | 已产出交付物 |
| 是否延迟 | 延迟判定结果 |
| 状态偏差 | 偏差程度判定 |

## 五、跨团队协调方法

### 5.1 团队分类与协调

从组织架构中学到：

**内部团队**：
- 需求团队、设计团队、开发团队、测试团队、实施团队

**外部团队**：
- 局方：市场部、云网部、全渠道部、政企部
- 供应商：IBOC、实业公司、承包助手

### 5.2 协调机制

**文档化协调**：
- 所有协调事项记录在问题跟踪表
- 明确责任人、解决期限、验证人
- 跟踪状态：Open/Close

**会议协调**：
- 内部评审：项目核心人员参与
- 客户评审：与市场、云网、全渠联合评审
- 转开发评审：业务方案开发+测试评审

### 5.3 外部依赖管理

从"关键外部依赖"表中学到：

| 字段 | 说明 |
|------|------|
| 依赖描述 | 需要外部支持的内容 |
| 完成时间 | 对方承诺的时间 |
| 对方单位 | 外部依赖方 |
| 对方接口人 | 具体对接人 |
| 负责人（我司） | 内部跟进人 |
| 状态 | 完成/进行中 |

**案例**：商机项目中的外部依赖：
- 承包助手接口对接
- 统一门户数据同步
- IBOC大模型API支持
- 实业短信通道

## 六、决策支持方法

### 6.1 信息汇总机制

**周报要素**（为决策提供信息基础）：
1. 项目总体进展（完成百分比+说明）
2. 本周完成情况
3. 下周工作计划
4. 问题与风险（待解决和协调事项）

### 6.2 决策建议框架

从实战案例中学到的**决策支持模式**：

**问题分类**：
- 高优先级：影响里程碑的关键问题
- 中优先级：影响效率的问题
- 低优先级：优化类问题

**建议输出方式**：
1. **风险预警**：提前识别，给出预防/应急方案
2. **资源申请**：明确所需资源及时间
3. **方案选择**：提供多方案对比分析
4. **进度预警**：里程碑偏差预警

### 6.3 数据驱动决策

**关键指标**：
- 整体进度百分比
- 里程碑完成率
- 风险级别分布
- 问题关闭率
- 缺陷密度

**案例**：项目周报中的进度计算公式：
```
总进度 = 方案设计进度×权重 + 系统开发进度×权重 + 系统测试进度×权重 + 系统试点进度×权重 + 系统推广进度×权重
```

## 七、实战案例启发

### 7.1 商机项目亮点

1. **多层计划分解**：从里程碑→总体计划→阶段计划，层次清晰
2. **实时进度跟踪**：每个任务都有"当前进展"状态
3. **风险前置管理**：提前识别12项风险，制定预防措施
4. **问题闭环管理**：17项问题跟踪，明确责任人和期限
5. **跨团队协作**：业务组、设计组、开发组、测试组并行

### 7.2 关键经验教训

**成功经验**：
- 需求阶段就与客户深度对齐（内部评审→客户评审→转开发评审）
- 技术风险提前验证（大模型能力、小程序审核）
- 资源依赖提前协调（IBOC、实业公司）

**待改进点**：
- 部分任务进度滞后（如里程碑4进度90%）
- 外部依赖存在不确定性（如实业短信接口变更）
- 需要加强跨系统数据映射（渠道视图、承包助手、智慧营维）
