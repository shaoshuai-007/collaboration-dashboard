## 一、总体方案设计方法

### 1.1 设计框架结构

从两份文档中总结出方案设计的标准框架：

```
1. 建设背景与目的
2. 总体概述
   ├── 业务目标
   ├── 功能架构
   └── 改造清单
3. 详细设计
   ├── 功能描述
   ├── 业务流程
   ├── 界面原型
   ├── 实现方案
   └── 物理模型
4. 接口协议
5. 非功能性设计
6. 附录
```

### 1.2 业务目标表述方式

**范例**（湖北电信商机平台）：
> 通过"统一标准、智能工具、闭环管控、数据赋能"，打造数字化运营新空间，实现商机管理从"线下分散"到"线上集中"、从"人工低效"到"智能高效"的转型，最终提升商机转化率、优化资源配置、增强市场竞争力。

**编写要点**：
- 明确设计理念（四个关键词）
- 描述转型路径（从X到Y）
- 指向具体目标（提升XX率）

## 二、技术架构描述方式

### 2.1 功能模块描述五步法

每个功能模块遵循统一描述模板：

```
1. 功能描述：一句话说明功能目标
2. 流程描述：业务流程图或步骤说明
3. 界面原型：UI设计稿或交互示意
4. 实现方案：详细的技术实现逻辑
5. 物理模型：涉及的数据库表设计
```

**范例**（商机隐私安全）：

| 章节 | 内容 |
|------|------|
| 功能描述 | 商机数据作为电信业务运营的核心资产，需构建完善的隐私保护机制 |
| 实现方案 | 用户登录后权限写入缓存；接口调用时匹配权限；属性脱敏规则配置 |
| 物理模型 | sys_role、sys_post、脱敏规则配置表 |

### 2.2 物理模型设计规范

**表设计包含要素**：
```sql
-- 示例：商机信息表
CREATE TABLE sale_opp_info (
    id              BIGINT PRIMARY KEY,
    sale_opp_id     VARCHAR(50),      -- 商机标识
    source_opp_id   VARCHAR(50),      -- 源系统商机唯一标识
    org_type        VARCHAR(10),      -- 商机归属类型
    org_id          VARCHAR(50),      -- 商机归属标识
    lifecycle_status VARCHAR(10),    -- 商机跟进状态
    collect_date    TIMESTAMP,        -- 采集时间
    data_period     VARCHAR(6),       -- 账期
    data_type       VARCHAR(10)       -- 数据类型
);
```

**字段约束说明**：
- 必填/选填明确标注
- 枚举值需列举完整
- 分区字段需说明分区策略

### 2.3 数据存储架构设计

**大数据量表分析模板**：

| 序号 | 表名 | 使用场景 | 数据量预估 |

## 三、系统集成设计（消息队列、接口）

### 3.1 消息队列设计规范

**消息队列申请清单格式**：

| 序号 | 名称 | 编码 | 生产方 | 消费方 | 用途 |
|------|------|------|--------|--------|------|
| 1 | 商机采集 | mkt_opp_collect | 外围系统 | 商机中台 | 商机汇聚 |
| 2 | 刷新标签 | mkt_opp_label | 商机中台 | 商机中台 | 标签计算通知 |
| 3 | 数据清洗 | mkt_opp_data_clean | 商机中台 | 商机中台 | 清洗流程触发 |

**消息格式示例**：
```json
{
    "transactionId": "123",
    "templateCode": "123",
    "appId": "xxx",
    "staffPhoneNumber": "xxx",
    "createDate": "2025-07-02 16:30:36",
    "sourceOppId": "12345678",
    "orgType": "1100",
    "orgId": "12345678",
    "lifecycleStatus": "1000",
    "dataType": "1000",
    "saleOppType": "1000",
    "oppAttrs": [
        {"attrSpecCode": "10001", "attrValue": "客户名称"},
        {"attrSpecCode": "10002", "attrValue": "075526744688"}
    ]
}
```

### 3.2 HTTP接口设计规范

**接口定义模板**：

```
接口名称：商机采集接口
接口描述：接收外部商机数据
接口提供方：商机运营管理平台
接口调用方：商机源系统
协议：HTTP/HTTPS
方法：POST
编码：UTF-8
```

**请求参数表**：

| 参数名称 | 参数编码 | 类型 | 必填 | 说明 |
|----------|----------|------|------|------|
| 交易流水号 | transactionId | String(100) | 是 | 不可重复 |
| 模板编码 | templateCode | String(100) | 是 | 双方商议 |
| 系统标识 | appId | String(100) | 是 | 调用方标识 |
| 商机属性列表 | oppAttrs | List\<Object\> | 是 | 商机属性集合 |

**响应参数表**：

| 参数编码 | 类型 | 说明 |
|----------|------|------|
| code | String | 0000成功，其他失败 |
| msg | String | 状态描述 |
| data | Object | 响应内容 |

### 3.3 接口安全设计

**签名机制**：
```
签名算法：HMAC-Sha256
签名规则：appId=xxx&body=xxx&timestamp=xxx&nonce=xxx
          （参数名ASCII码升序排列）
```

**加密机制**：
```
加密算法：AES-256-CBC
密钥：联调时提供
```

### 4.2 数据清洗规则设计

**清洗规则示例**：

| 序号 | 规则名称 | 清洗描述 |
|------|----------|----------|
| 1 | 身份证号码 | 清除非数字字符；18位长度校验；格式校验 |
| 2 | 手机号码 | 移除非数字；11位纯数字；1开头，第二位3-9 |
| 3 | 日期 | 清除空格；统一转换为YYYY-MM-DD格式 |
| 4 | 地址 | 长度≥6；至少包含4个汉字 |

### 4.3 数据归档策略

**归档规则**：
- 数据保留周期：2年
- 归档时机：每日0时执行
- 归档存储：数据湖

**分区分表策略**：
```
商机信息表：按月分区 + 商机标识取模（10个分区）
商机属性表：同商机表
去重校验码表：校验码HASH值取模（10个分区）
```

```json

}

// 出参

    "code": "0000",
    "text": "识别的文本内容",
    "confidence": 0.95
}
```

**文本属性提取接口**：
```json

    "attrs": [

### 5.4 异步处理机制

**任务队列设计**：
```
1. 定时任务扫描待处理记录

3. 执行AI识别任务
4. 更新识别状态（成功/失败）
5. 失败重试机制（最多3次）

## 六、文档编写规范总结

### 6.1 文档结构规范

```
文档说明
├── 编制说明
├── 文档级别（四级文档）
├── 适用范围
├── 术语定义
└── 起草单位

主体内容
├── 需求描述/建设背景
├── 总体方案
│   ├── 业务目标
│   ├── 方案概述
│   └── 功能架构
├── 详细设计
│   ├── 功能描述
│   ├── 流程描述
│   ├── 界面原型
│   ├── 实现方案
│   └── 物理模型
├── 接口协议
├── 消息队列申请清单
└── 非功能性设计

附录
├── 数据清洗规则
├── 字典数据
└── 编码规范
```

### 6.2 图表使用规范

**常用图表类型**：
- 系统架构图：展示系统整体架构
- 数据流图：展示数据流转过程
- 流程图：展示业务流程或技术流程
- 物理模型图：展示数据库表结构

**图表要求**：
- 每个图表需有标题和编号
- 图表需有文字说明
- 复杂图表需分层次展示

### 6.3 接口文档规范

**必须包含内容**：
1. 接口基本信息（名称、描述、协议、方法）
2. 接口认证方式
3. 请求参数表（名称、编码、类型、必填、说明）
4. 响应参数表
5. 报文样例（入参、出参）
6. 响应编码说明

### 6.4 状态机/枚举设计

**枚举值定义模板**：
```
状态编码 | 状态名称 | 说明
1000    | 待联系  | 录入商机初始状态
1100    | 跟进中  | 已联系客户
1200    | 已转化  | 客户已办理
1300    | 已关闭  | 客户无意向
```

## 七、学习心得

### 7.1 架构设计要点

1. **分层解耦**：前后端分离、数据层独立、接口标准化
2. **扩展性设计**：任务队列可配置并行数、规则引擎可配置
3. **容错机制**：失败重试、异常处理、数据归档
4. **安全设计**：权限管控、数据脱敏、接口加密

### 7.2 文档编写技巧

1. **结构清晰**：从总到分、从概念到实现
2. **图文结合**：架构图+流程图+数据模型图
3. **示例丰富**：接口报文样例、枚举值示例
4. **表格规范**：参数表、字段表、规则表

### 7.3 实践建议

- 需求分析阶段：明确业务目标、梳理功能清单
- 架构设计阶段：画系统架构图、数据流图、物理模型
- 接口设计阶段：定义协议、参数、报文格式
- 实施阶段：按时间节点分模块开发
