## 二、数据描述规范

### 2.1 数据描述内容要求

需求规格说明书中的数据描述包含以下要素：

| 要素 | 说明 |
|------|------|
| 涉及的表、字段 | 明确数据存储位置 |
| 数据处理逻辑 | 数据的增删改查规则 |
| 数据权限说明 | 查看权限、操作权限 |
| 数据模型设计 | 新建表、修改表字段 |

### 2.2 数据模型设计规范

**表结构设计格式**：

```
表名 `table_name` (新建/改造/按xxx分表)

| 字段 | 名称 | 数据类型 | 备注 |
|------|------|---------|------|
| field_name | 字段中文名 | 数据类型 | 说明 |
```

**设计要点**：
- 明确表的操作类型（新建/改造）
- 分表策略需注明分表规则
- 字段备注需包含业务含义说明
- 状态类字段需注明状态值对应关系

### 2.3 系统设计师的衔接要点

> **需求规格 → 系统设计的衔接**

| 需求规格内容 | 系统设计产出 |
|-------------|-------------|
| 数据模型设计 | 详细数据库设计文档 |
| 数据权限说明 | 权限模型设计 |
| 数据处理规则 | 业务逻辑设计 |

**我的职责**：
1. 将需求中的数据描述转化为完整的数据库设计
2. 补充索引、约束、存储过程等数据库对象设计
3. 设计数据迁移方案（改造项目）
4. 完善数据字典

## 三、接口需求定义

### 3.1 接口清单

需求规格需列出所有外部交互接口：
- 我方提供的接口
- 外围提供的接口

### 3.2 单个接口定义规范

| 项目 | 内容要求 |
|------|---------|
| 接口名称 | 明确功能命名 |
| 提供方 | 明确接口归属方 |
| 接口描述 | 接口用途说明 |
| 接口方式 | 同义词/直连数据库/协议调用 |
| 接口参数 | 入参、出参、数据类型、封装方式 |
| 异常处理 | 异常信息定义 |

### 3.3 接口参数定义要点

- **入参**：字段名、类型、必填性、取值范围
- **出参**：返回码、返回信息、数据结构
- **数据封装**：JSON/XML格式定义
- **参考案例**：提供示例数据

### 3.4 系统设计师的衔接要点

> **需求规格 → 系统设计的衔接**

| 需求规格内容 | 系统设计产出 |
|-------------|-------------|
| 接口清单 | 接口详细设计文档 |
| 接口参数 | API接口规范（Swagger/OpenAPI） |
| 接口方式 | 集成架构设计 |
| 异常处理 | 异常处理机制设计 |

**我的职责**：
1. 将接口需求转化为详细的API设计
2. 定义完整的接口协议（REST/RPC）
3. 设计接口安全机制（鉴权、加密）
4. 编写接口Mock数据用于联调
5. 评估接口性能需求

## 四、非功能需求

### 4.1 安全性需求

需求规格需考虑三类安全策略：

| 类型 | 内容 |
|------|------|
| 业务安全 | 防止F12调试、页面篡改 |
| 数据安全 | 加密、脱敏、数据备份 |
| 访问安全 | SQL注入防护、身份鉴权、权限控制 |

### 4.2 性能需求

**系统整体性能指标**：

| 指标 | 示例要求 |
|------|---------|
| 在线用户数 | 200万 |
| 平稳运行时间 | 7×24小时 |
| 平均响应时间 | <2秒 |
| CPU使用率 | <75% |

**功能/接口性能指标**：

按功能重要性分类定义：
- **高频功能**：使用频率高的功能
- **关键功能**：登录、注册、支付等核心功能
- **特色功能**：产品卖点功能
- **大数据量功能**：报表查询等

**示例格式**：
```
登录功能：
- 并发用户数：500
- 响应时间：2s
- TPS：500/s
- CPU：<75%
```

### 4.3 网络需求

需考虑的网络场景：
- 内网访问
- 外网访问
- VPN网络
- 特殊网段要求

### 4.4 兼容性需求

明确需要兼容的：
- 浏览器类型和版本
- 手机机型和系统版本
- 屏幕分辨率

### 4.5 系统设计师的衔接要点

> **需求规格 → 系统设计的衔接**

| 需求规格内容 | 系统设计产出 |
|-------------|-------------|
| 安全性需求 | 安全架构设计、安全方案 |
| 性能需求 | 性能优化方案、容量规划 |
| 网络需求 | 网络架构设计、部署方案 |
| 兼容性需求 | 前端适配方案、测试矩阵 |

**我的职责**：
1. 将性能指标转化为架构设计约束
2. 设计缓存、异步处理等性能优化方案
3. 设计安全架构和防护措施
4. 评估系统容量，规划硬件资源
5. 制定部署架构方案

## 五、流程图类型选择

### 5.1 流程图类型对照表

| 图类型 | 适用场景 | 侧重点 |
|-------|---------|-------|
| 流程图 | 系统内的业务规则判定、处理分支逻辑 | 系统内部逻辑 |
| 泳道图 | 系统之间有交互，重点突出业务判定逻辑 | 跨系统协作 |
| 时序图 | 系统之间交互处理环节多，重点突出交互逻辑 | 交互时序 |
| 数据逻辑图 | 重点突出不同环节的数据处理和保存差异 | 数据流转 |
| 状态变更流转图 | 多个状态变更的处理逻辑 | 状态机设计 |

### 5.2 选择决策树

```
需要画流程图吗？
├── 只涉及单个系统？
│   └── 是 → 流程图（关注内部逻辑）
│   └── 否 → 涉及多个系统交互？
│       ├── 交互环节多？→ 时序图（关注交互顺序）
│       └── 业务判定复杂？→ 泳道图（关注职责分工）
│
├── 重点展示数据处理？
│   └── 是 → 数据逻辑图（关注数据变化）
│
└── 重点展示状态变化？
    └── 是 → 状态变更流转图（关注状态流转）
```

### 5.3 系统设计师的衔接要点

> **需求规格 → 系统设计的衔接**

| 需求规格流程图 | 系统设计产出 |
|--------------|-------------|
| 流程图 | 详细业务逻辑设计 |
| 泳道图 | 系统集成设计、职责划分 |
| 时序图 | 接口调用顺序设计、超时处理 |
| 数据逻辑图 | 数据库设计、数据流转设计 |
| 状态变更流转图 | 状态机设计、状态迁移表 |

**我的职责**：
1. 将需求流程图转化为详细的系统设计流程
2. 补充异常分支和技术细节
3. 设计技术层面的时序和交互
4. 明确数据在各环节的处理规则

## 七、从需求规格到系统设计的衔接

### 7.1 衔接映射表

| 需求规格章节 | 系统设计产出 | 工尺职责 |
|-------------|-------------|---------|
| 数据描述 | 数据库设计文档 | 详细表结构、索引、存储过程 |
| 接口需求 | 接口设计文档 | API规范、调用链路、异常处理 |
| 安全性需求 | 安全设计方案 | 安全架构、防护措施 |
| 性能需求 | 性能优化方案 | 缓存设计、异步处理、容量规划 |
| 流程描述 | 详细设计流程图 | 技术流程、异常分支、时序设计 |

### 7.2 需求评审关注点

作为系统设计师，评审需求规格时需关注：

1. **数据完整性**
   - 字段定义是否完整
   - 数据类型是否合理
   - 分表策略是否明确

2. **接口完整性**
   - 接口清单是否完整
   - 参数定义是否明确
   - 异常处理是否覆盖

3. **非功能需求可行性**
   - 性能指标是否可达成
   - 安全需求是否可实现
   - 资源需求是否可满足

4. **流程完整性**
   - 异常流程是否覆盖
   - 边界条件是否明确
   - 技术可行性是否评估

## 八、学习心得

通过学习天源迪科需求规格说明书模板，我作为系统设计师理解到：

1. **需求规格是系统设计的输入**，需要在设计前确保需求规格的完整性和清晰性

2. **数据描述需要结构化**，表结构设计格式为后续数据库设计提供了基础

3. **接口需求需要标准化**，明确出入参和异常处理有助于设计阶段的接口规范制定

4. **非功能需求需要量化**，具体的性能指标是架构设计的重要约束条件

5. **流程图类型选择有章可循**，不同场景选择不同类型，侧重点各有不同
