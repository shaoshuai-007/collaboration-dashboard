## 📌 核心认知：需求规格与方案设计的关系

作为方案设计师，我需要理解：**需求规格说明书是方案的"地基"，方案PPT是需求价值的"呈现载体"**。

```
需求规格 → 方案设计的衔接点：
├─ 业务场景 → PPT中的"业务痛点/场景描述"章节
├─ 业务流程 → PPT中的"业务流程优化"章节（可视化呈现）
├─ 功能结构 → PPT中的"功能架构图"
├─ 功能清单 → PPT中的"功能亮点/特色功能"提炼
└─ 非功能需求 → PPT中的"技术优势/性能保障"章节
```

## 一、业务需求描述规范

### 1.1 业务场景编写规范

**核心方法：5W1H分析法**

| 要素 | 含义 | 方案设计应用 |
|------|------|-------------|
| Who（谁） | 用户角色 | PPT中明确目标用户画像 |
| When（何时） | 触发时机 | 场景化描述的切入点 |
| Where（何地） | 使用场景 | 业务场景配图素材选择依据 |
| What（何事） | 业务内容 | 功能描述的核心内容 |
| Why（为何） | 业务价值 | **PPT核心价值主张提炼来源** |
| How（如何） | 实现方式 | 技术方案要点 |

**方案设计启示**：
- 业务场景等同于"用户故事"，是PPT故事线的基础
- 5W1H法可以帮助我快速提炼业务场景的核心要素
- **Why（业务价值）是PPT价值主张的关键来源**

### 1.2 业务流程编写规范

**要点**：
- 从业务角度出发，描述业务场景流程
- 侧重点：不同角色的操作场景
- **必须与业务方确认流程**

**方案设计启示**：
- 业务流程图是PPT中"现状流程vs优化流程"对比的核心素材
- 不同角色视角有助于PPT中"多角色收益"章节的编写

### 1.3 现状分析（改造需求必选）

**内容要求**：
- 业务现状：目前的界面、业务流程、逻辑等
- 目前问题：**可用对比图呈现"现状vs目标"**

**方案设计启示**：
- "现状vs目标"对比图是PPT中最直观的展示方式
- 痛点问题是PPT中"问题分析"章节的核心内容

## 二、功能需求描述规范

### 2.1 功能结构编写规范

**要求**：结构图，描述整体需求中所有功能模块架构关系

**方案设计启示**：
- 功能结构图 = PPT中的"功能架构图/系统架构图"
- 架构图设计要点：
  - 层次清晰（系统层→模块层→功能层）
  - 模块关系明确
  - 重点突出核心功能模块

### 2.2 功能清单编写规范

**表格格式**：

| 功能模块 | 功能模块概述 | 使用角色 | 改造 | 优先级 |
|---------|-------------|---------|------|--------|
| | | | 新建or改造 | |

**方案设计启示**：
- 功能清单可提炼为PPT中的"功能概览"页面
- **优先级字段**帮助识别PPT中需要重点突出的核心功能
- **使用角色字段**帮助提炼不同用户的收益点

### 2.3 功能需求详情结构

每个功能点包含以下板块：

```
4.3.XXX功能需求
├─ 4.3.1 功能描述 → 功能说明
├─ 4.3.2 流程描述 → 流程图/泳道图/时序图
├─ 4.3.3 交互原型图 → 页面交互流程+原型地址
├─ 4.3.4 业务描述 → 页面元素+业务规则
├─ 4.3.5 数据描述 → 数据表/字段/权限
└─ 4.3.6 影响范围 → 改造影响+风险分析
```

**方案设计启示**：
- 功能描述是PPT功能详情页的内容基础
- 流程图和交互原型图可直接用于PPT
- 影响范围分析有助于PPT中的"风险控制"章节

## 三、流程图类型选择规范

### 3.1 流程图选择矩阵（重点）

| 图类型 | 适用场景 | 方案设计应用 |
|-------|---------|-------------|
| **流程图** | 系统内的业务规则判定、处理分支逻辑 | PPT中单一系统流程展示 |
| **泳道图** | 系统之间有交互，重点突出业务判定逻辑 | PPT中跨系统协作流程 |
| **时序图** | 系统之间交互处理环节多，重点突出交互逻辑 | PPT中技术实现展示 |
| **数据逻辑图** | 重点突出不同环节的数据处理和保存差异 | PPT中数据处理流程 |
| **状态变更流转图** | 多个状态变更的处理逻辑 | PPT中业务状态流转 |

**方案设计启示**：
- 选择正确的流程图类型，能让PPT更清晰地传达业务逻辑
- **泳道图**是跨系统场景的最佳选择（突出角色分工）
- 流程图要包含异常流程和异常处理要求

### 3.2 流程图设计要点

1. 注重在系统实现层面
2. 关注补充异常流程和异常处理要求
3. 与业务方确认流程

## 四、交互原型图设计要求

### 4.1 核心要求

1. **多个界面需要包含页面交互流程**
2. **可附带原型在线访问地址**

**方案设计启示**：
- 交互原型图是PPT中"界面展示"章节的核心素材
- 页面交互流程可以帮助梳理用户操作路径
- 原型地址可作为PPT附录或演示链接

### 4.2 交互原型设计规范（推断）

基于模板内容，交互原型图应包含：
- 页面布局结构
- 核心操作元素
- 页面间跳转关系
- 异常状态处理

## 五、方案设计师的实践建议

### 5.1 从需求规格提炼PPT内容的方法

| 需求规格章节 | PPT对应章节 | 提炼方法 |
|-------------|------------|---------|
| 建设背景 | 项目背景 | 压缩为一页，突出核心背景 |
| 建设目的 | 项目目标 | 分点列出，量化指标 |
| 业务场景 | 业务场景描述 | 5W1H提炼核心场景 |
| 业务流程 | 业务流程优化 | 现状流程vs优化流程对比 |
| 功能结构 | 功能架构 | 转化为架构图 |
| 功能清单 | 功能概览 | 表格化展示 |
| 非功能需求 | 技术优势 | 量化指标展示 |
| 待确定问题 | 风险控制 | 风险分析与应对措施 |

### 5.2 PPT设计关注点

1. **价值主张**：从业务场景的Why要素提炼
2. **功能亮点**：从功能清单的优先级字段识别
3. **流程优化**：从业务流程的"现状vs目标"对比
4. **技术优势**：从非功能需求的量化指标转化
5. **风险控制**：从影响范围分析提炼

### 5.3 质量检查要点

作为方案设计师，我需要在拿到需求规格时检查：
- [ ] 业务场景是否使用5W1H分析
- [ ] 业务流程是否与业务方确认
- [ ] 功能描述是否完整
- [ ] 流程图类型选择是否正确
- [ ] 数据模型设计是否完整
- [ ] 非功能需求是否量化
- [ ] 待确定问题是否已闭环

## 六、学习总结

### 核心收获

1. **5W1H法**是业务场景分析的标准方法，也是PPT价值主张提炼的利器
2. **流程图选择**要根据场景特点选择合适类型，泳道图适合跨系统协作
3. **功能结构图**是PPT架构图的直接来源
4. **"现状vs目标"对比**是PPT中最直观的展示方式
5. **非功能需求的量化指标**是PPT技术优势展示的核心内容
