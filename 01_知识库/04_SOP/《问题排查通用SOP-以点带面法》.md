# 问题排查通用SOP - 以点带面法

## 核心思维

> **发现一个问题，不是只修这一个，而是检查所有同类项是否都有同样问题。**

---

## 排查流程（五步法）

### 第一步：确认问题现象
- 问题是什么？
- 预期行为是什么？
- 实际行为是什么？

### 第二步：定位问题根源
- 逐层排查：配置 → 执行 → 结果
- 使用工具检查状态（如 `openclaw cron list`、`openclaw cron runs`）
- 查看日志或运行历史

### 第三步：修复单个问题
- 针对根源制定修复方案
- 执行修复
- 验证修复效果

### 第四步：举一反三（关键！）
- **同类项检查**：是否存在其他类似配置/任务/项？
- **批量排查**：检查所有同类项是否有同样问题
- **批量修复**：如有问题，一并修复

### 第五步：沉淀方法论
- 记录问题原因和修复方法
- 更新SOP或知识库
- 形成检查清单，预防复发

---

## 实战案例：定时提醒未推送问题

### 问题现象
用户反馈定时提醒未触发/未收到

### 排查过程
| 步骤 | 动作 | 结果 |
|:----:|------|------|
| 1 | 确认任务存在 | ✅ 两个任务都存在 |
| 2 | 检查触发状态 | ✅ 任务已触发（runs显示ok） |
| 3 | 查看触发结果 | ⚠️ 生成了内容，但未推送 |
| 4 | 分析根源 | ❌ 缺少 `--channel qqbot --to` 参数 |
| 5 | 修复任务1 | ✅ 添加推送配置 |
| 6 | **举一反三** | ✅ 检查任务2，同样缺少配置 |
| 7 | 批量修复 | ✅ 两个任务都已修复 |

### 修复命令
```bash
openclaw cron edit <job_id> --channel qqbot --to <openid> --announce
```

### 沉淀成果
- 创建定时提醒时，必须包含 `--channel` 和 `--to` 参数
- 已写入MEMORY.md作为操作规范

## 适用场景

| 场景 | 同类项示例 |
|------|------------|
| 配置问题 | 多个定时任务、多个推送渠道、多个环境配置 |
| 代码缺陷 | 同类函数、相似逻辑、同一模块的多处调用 |
| 流程漏洞 | 多个审批节点、多个检查环节、多个系统对接 |
| 数据异常 | 同类指标、相似数据源、同一批次的数据 |

---

## 配置同步专项检查

### 触发条件
- 获取新的用户信息（openid、邮箱等）
- 创建新的定时任务
- 修改推送配置

### 检查清单
- [ ] MEMORY.md已更新
- [ ] 定时任务已配置推送参数（--channel, --to）
- [ ] 手动测试推送成功
- [ ] 检查cron运行日志确认执行成功

---

## 检查清单模板

```markdown
## [问题名称] 排查检查清单

- [ ] 问题现象确认
- [ ] 根源定位
- [ ] 单点修复验证
- [ ] 同类项A检查：____（状态：正常/异常）
- [ ] 同类项B检查：____（状态：正常/异常）
- [ ] 同类项C检查：____（状态：正常/异常）
- [ ] 批量修复（如需要）
- [ ] 方法论沉淀
```
