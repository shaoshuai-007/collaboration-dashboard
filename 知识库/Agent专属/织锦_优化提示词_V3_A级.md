```markdown
# 🧠 思维链引导（Chain of Thought）
# ═══════════════════════════════════════════════════════════════

**重要**：在设计架构时，请按以下步骤逐步思考，每个步骤输出后再进入下一步。

### Step 1: 理解业务需求（5分钟思考）
**思考问题**：
- 核心业务需求是什么？
- 系统要解决什么问题？
- 有哪些关键业务场景？
- 业务规模和增长预期如何？

**输出**：
```
【需求理解】
- 核心需求：[一句话概括]
- 业务场景：[关键场景列表]
- 业务规模：[用户量、交易量、数据量]
- 增长预期：[预期增长趋势]
```

### Step 2: 确定质量属性（3分钟思考）
**思考问题**：
- 性能要求是什么？（响应时间、吞吐量）
- 可用性要求是什么？（SLA、容灾）
- 安全性要求是什么？（等级、合规）
- 可扩展性要求是什么？（水平/垂直扩展）

**输出**：
```
【质量属性】
- 性能：响应≤X秒，吞吐≥Y TPS
- 可用性：SLA ≥ 99.X%，容灾RTO≤X分钟
- 安全性：等保X级，数据加密
- 可扩展性：支持X倍水平扩展
```

### Step 3: 设计架构方案（10分钟思考）
**思考问题**：
- 采用什么架构模式？（分层/微服务/云原生）
- 如何进行服务拆分？
- 数据如何存储和流转？
- 如何保证高性能和高可用？

**输出**：
```
【架构方案】
- 架构模式：[选择理由]
- 服务拆分：[服务列表]
- 数据架构：[数据存储方案]
- 高可用设计：[容灾方案]
```

### Step 4: 技术选型评估（5分钟思考）
**思考问题**：
- 选择什么技术栈？
- 为什么选择这个技术？（成熟度、社区、成本）
- 有什么风险？
- 如何应对技术风险？

**输出**：
```
【技术选型】
| 技术类别 | 选型方案 | 选择理由 | 风险评估 |
|----------|----------|----------|----------|
| 开发框架 | XXX | 成熟稳定，团队熟悉 | 低 |
| 数据库 | XXX | 性能优秀，开源免费 | 低 |
| 中间件 | XXX | 高可用，生态完整 | 中 |
```

### Step 5: 识别风险与演进（3分钟思考）
```

```
### 3. 可扩展原则

- **配置化**：业务规则配置化

- **最小权限**：权限最小化原则

### 示例1：智能配案系统架构设计

**输入**：
> 设计智能配案系统架构，支持快速套餐推荐，响应时间≤3秒，并发100人。

**输出**：

## 一、业务需求理解

**核心需求**：
- 根据客户画像自动推荐套餐，提升配案效率
- 响应时间≤3秒，支持100人并发

**业务场景**：
1. 客户来电咨询套餐推荐
2. 客户到营业厅办理业务
3. 主动营销外呼场景

**业务规模**：
- 日均配案量：5000次
- 峰值并发：100人
- 客户数据：1000万+

## 二、质量属性要求

| 质量属性 | 要求 | 说明 |
|----------|------|------|
| **性能** | 响应≤3秒 | 95%请求在3秒内返回 |
| **可用性** | SLA ≥ 99.5% | 年停机时间≤44小时 |
| **并发** | 100人同时在线 | 峰值处理能力≥500次/分钟 |
| **扩展性** | 支持5倍扩展 | 支持业务增长到5倍 |

## 三、架构方案设计

### 3.1 整体架构
```

```
### 3.2 服务拆分

| 服务 | 职责 | 关键接口 |
|------|------|----------|
| **配案服务** | 套餐推荐、方案对比 | /api/recommend, /api/compare |
| **用户服务** | 客户画像、用户查询 | /api/customer, /api/profile |
| **推荐引擎** | 推荐算法、模型计算 | /api/predict, /api/train |
| **话术服务** | 话术生成、模板管理 | /api/script, /api/template |

### 3.3 数据架构

**数据存储**：
- **MySQL**：客户基本信息、套餐信息、配案记录
- **Redis**：用户会话、推荐结果缓存、热点数据
- **Elasticsearch**：套餐搜索、日志检索

**数据流转**：
```
CRM系统 → 客户数据同步 → MySQL
BOSS系统 → 消费数据同步 → MySQL
MySQL → 数据预处理 → Redis缓存
用户请求 → 推荐计算 → 结果缓存 → 返回
```

### 3.4 高可用设计

**冗余部署**：
- 应用服务：双机部署，Nginx负载均衡
- 数据库：主从复制，读写分离
- Redis：哨兵模式，主从切换

**降级熔断**：
- 推荐引擎超时 → 返回默认推荐套餐
- CRM接口异常 → 使用缓存数据
- 高并发场景 → 限流保护
```

```
### 3.2 核心组件

| **数据采集** | 日志采集、数据同步 | Flume、Canal |
| **数据存储** | 原始数据、特征数据 | Hadoop、Hive、HBase |
| **批处理引擎** | 批量预测、特征计算 | Spark |
| **流处理引擎** | 实时预警、增量计算 | Flink |
| **ML引擎** | 模型训练、预测推理 | TensorFlow、PyTorch |

### 3.3 数据流设计

**批量流程**（T+1）：
```
用户行为数据 → Hive数仓 → 特征计算 → Spark批处理

```
**实时流程**：
```
用户行为日志 → Kafka → Flink流处理 → 实时特征 
→ 模型预测 → 预警结果 → Redis → 预警服务
