## 二、里程碑计划编制方法

### 2.1 里程碑定义原则

根据模板说明：
1. **每月至少1个里程碑**（含初验、终验），项目结项视为里程碑计划制定完整
2. **里程碑计划与完成必须有明确的交付物**
3. **里程碑计划按功能对标一级功能模块制定**

### 2.2 字段定义

| 字段名 | 说明 | 示例 |
|--------|------|------|
| 序号 | 里程碑编号 | 1, 2, 3... |
| 里程碑 | 里程碑名称 | 项目启动、需求分析完成、系统上线 |
| 计划交付物 | 预期输出 | 项目启动材料、需求规格说明书 |
| 完成标准 | 验收条件 | 完成项目团队创建，明确项目里程碑计划 |
| 计划完成时间 | 截止日期 | 日期格式 |
| 进度% | 当前进度 | 0-100%或小数 |
| 进度说明 | 进展描述 | 具体完成情况说明 |
| 实际完成时间 | 完成日期 | 实际完成日期 |
| 实际工时(天) | 实际耗时 | 天数 |
| 实际输出物 | 实际产出 | 交付物名称 |
| 提交SVN地址 | 交付物位置 | 文档存放路径 |
| 是否延迟 | 自动计算 | 基于公式判断 |
| 状态偏差 | 自动计算 | 正常/可控/异常 |

### 2.3 延迟判断逻辑

**是否延迟判断**：
- 当进度不为100%，且计划完成时间 >= 当天 → 非延迟
- 当进度为100%，且实际完成时间 <= 计划完成时间 → 非延迟
- 其余情况 → 延迟

**状态偏差判断**：
- 无偏差（正常）：进度为100% 或 当天 <= 计划完成时间
- 可控：偏差 <= 15%，计算：(当天-计划完成时间)/实际工时 <= 15%
- 异常：偏差 > 15%

### 2.4 实战案例里程碑示例

商机统一运营管理项目共设置11个里程碑：
1. 项目启动
2. 项目立项
3. 商机笔记本小程序注册及传统商机录入、查询功能开发及上线
4. 商机分类及要素管理、拍照AI识别录入、商机提醒及查询功能开发及上线
5. 商机采集模板配置、商机分享、商机汇聚、传统规则清洗功能开发及上线
6. 随手录拍照+语音AI识别录入、商机增强、商机打标、商机运营统计视图功能开发及上线
7. 实时作战大屏、商机总览视图功能开发及上线
8. 随手录视频AI识别录入、商机对外赋能功能开发及上线
9. 执行跟踪视图、效能评估视图功能开发及上线
10. 项目初验
11. 项目终验

**特点分析**：
- 里程碑按功能模块划分，而非时间阶段
- 每个里程碑有明确的完成标准和交付物
- 从项目启动到终验形成完整闭环

## 三、总体计划编制方法

### 3.1 两种编制模式

**样例1（先横后纵）**：以里程碑为主轴，逐级展开
- 里程碑任务名称 → 关键目标/任务 → 详细任务
- 适合：里程碑导向的项目，强调阶段性交付

**样例2（先纵后横）**：以阶段为主轴，按阶段分解
- 任务目标 → 阶段 → 一级任务 → 二级任务
- 适合：阶段明显的项目，强调过程管控

### 3.2 总体计划字段定义

| 字段名 | 说明 |
|--------|------|
| 序号 | 任务编号 |
| 里程碑任务名称 | 所属里程碑 |
| 关键任务 | 里程碑对应的关键目标或任务 |
| 详细任务 | 具体执行任务 |
| 主要工作内容/备注 | 任务描述 |
| 是否里程碑 | 标记是否为里程碑节点 |
| 前置任务 | 任务依赖关系 |
| 输出物 | 任务产出 |
| 当前进展 | 进行中/已完成等状态 |
| 计划开始时间 | 开始日期 |
| 计划完成日期 | 结束日期 |
| 实际完成日期 | 实际完成日期 |
| 进度 | 0-1或百分比 |
| 责任人 | 任务负责人 |
| 备注 | 补充说明 |

### 3.3 任务分解层次

```
里程碑（一级）
  └── 关键任务（二级）
        └── 详细任务（三级）
              └── 工作内容描述
```

### 3.4 实战案例任务分解示例

以"里程碑一：商机笔记本小程序注册及传统商机录入、查询功能开发及上线"为例：

```

│   ├── 业务方案输出 → 输出：业务方案

│   └── 详细设计输出 → 输出：详细设计
├── 版本开发
│   ├── 商机笔记本-快捷登录
│   ├── 商机笔记本-入口及首页
│   ├── 商机笔记本-采集模板管理
│   └── ...

    ├── 功能测试
    └── 回归测试
```

## 四、进度跟踪方法

### 4.1 进度记录方式

1. **进度百分比**：用数值0-1或百分比表示
2. **当前进展状态**：已完成/进行中/未开始
3. **实际完成日期**：记录任务实际完成时间
4. **进度说明**：详细描述当前进展情况

### 4.2 进度更新频度

- 里程碑计划：**每周更新至少1次**
- 总体计划：**按周/日更新**
- 项目周报：**按周汇报**

### 4.3 进度偏差计算

**偏差公式**：
```
偏差 = (当天 - 计划完成时间) / 实际工时
```

- 偏差 > 15% → 异常（红色预警）
- 偏差 ≤ 15% → 可控（黄色警示）
- 无偏差 → 正常（绿色）

### 4.4 实战案例进度跟踪特点

案例中详细记录了进度变更历史，如：
```
8/4：渗透测试漏洞已修复...
8/11：合同已签订，待云网下单...
8/14：公众已下单，订单审批中...
8/22：安全订单流程仍在审批中...
8/28：安全扫描订单审批完成...
```

**优点**：形成了完整的进度追溯链，便于复盘和风险预警。

## 五、风险管控方法

### 5.1 风险识别框架

模板提供了风险识别指引清单（27项问题），覆盖：
- 需求风险：稳定性、完整性、清晰性、合理性、可行性
- 设计风险：功能性、难度、接口、性能、易测性
- 工程风险：可维护性、可靠性、安全性
- 管理风险：过程管理、组织、经验、人员

### 5.2 风险跟踪表字段

| 字段名 | 说明 |
|--------|------|
| 序号 | 风险编号 |
| 类别 | 风险/机遇 |
| 风险/机遇描述 | 分析总结后的描述 |
| 类型 | 收入类/进度类/需求类/政策类/成本类/资源类 |
| 发生概率 | 极有可能发生/有可能发生/不太可能发生 |
| 影响程度 | 影响大/影响中/影响小 |
| 发生时机 | 识别日期起X周以内 |
| 级别 | 高/中/低 |
| 预防/创造措施 | 预防措施（风险）或创造措施（机遇） |
| 应急/利用措施 | 应急措施（风险）或利用措施（机遇） |
| 识别日期 | 发现日期 |
| 对策负责人 | 负责解决的人 |
| 状态 | 已关闭/跟踪中/已发生 |
| 跟踪日志 | 详细的跟踪记录 |

### 5.3 风险级别判定规则

- **高风险**：级别高的必须制定预防措施和应急措施
- **中风险**：需要持续跟踪
- **低风险**：记录备案

### 5.4 实战案例风险识别示例

| 风险描述 | 类型 | 概率 | 影响 | 级别 | 预防措施 |
|----------|------|------|------|------|----------|
| 6月上线商机笔记本功能，时间紧任务重，存在延期风险 | 进度类 | 极有可能 | 影响中 | 高 | 与客户沟通分两版本上线 |
| 复用分销注册小程序风险 | 需求类 | 极有可能 | 影响大 | 高 | 确认小程序合并方案，明确运营单位 |
| 小程序审核风险 | 政策类 | 极有可能 | 影响大 | 高 | 提前发起小程序变更审核申请 |
| 微信手机号识别收费问题 | 成本类 | 极有可能 | 影响中 | 高 | 评估替代方案，确认费用承担方 |

**案例特点**：
- 风险识别细致，覆盖进度、需求、政策、成本、资源等多维度
- 每条风险都有明确的预防措施和跟踪日志
- 风险状态动态更新（已关闭/跟踪中）

## 六、周报编写方法

### 6.1 周报核心要素

根据模板要求，周报必须包含以下要素：
1. **项目总体进展**：整体进度百分比和阶段完成情况
2. **本期重点完成情况**：本周主要成果
3. **下周工作计划**：下周预期工作
4. **问题与风险**：待解决和协调事项

### 6.2 周报字段定义

| 字段名 | 说明 |
|--------|------|
| 项目名称 | 项目全称 |
| 责任人 | 局方+乙方负责人 |
| 完成进度 | 项目总体进度百分比 |
| 偏差 | 正常/延迟 |
| 项目总体进展(*) | 总体进展描述，包含各阶段完成百分比 |
| 本周完成情况(*) | 本周具体完成事项 |
| 下周工作计划 | 下周计划工作 |
| 问题与风险（待解决和协调事项）(*) | 问题、风险及需要协调事项 |
| 备注 | 其他补充 |

### 6.3 实战案例周报示例分析

案例中的周报格式：
```
项目名称：2022年安徽电信客户管理中心加密升级改造项目
责任人：局方：XXX，迪科：XXX
完成进度：29.5%
偏差：正常

项目总体进展：
一、总体进展（完成29.5%）：
    1、方案设计（已完成）
    2、系统开发（完成30%）
    3、系统测试（完成5%）
    4、系统试点（未开始）
    5、系统推广（未开始）

本周完成情况：
    1、立项汇报已完成
    2、敏感信息加密存储方案设计已完成
    3、统一使用人改造各中心开发已完成
    ...

下周工作计划：
    1、统一使用人去698/699属性写入版本合并主干测试
    2、量子加密方案的功能验证

问题与风险：
    [详细描述技术问题和风险点]
```

### 6.4 周报编写要点

1. **量化进展**：用百分比和具体数字描述
2. **结构清晰**：按阶段或模块分类汇报
3. **问题突出**：问题与风险单独列出，便于协调
4. **闭环管理**：下周计划与本周问题形成呼应

## 七、实战案例分析（对比模板）

### 7.1 结构对比

| 维度 | 模板设计 | 实战案例 | 差异分析 |
|------|----------|----------|----------|
| Sheet数量 | 17个 | 13个 | 精简了非必要Sheet |

| 风险记录 | 模板示例1条 | 实际记录7条+ | 覆盖全面 |

### 7.2 字段应用对比

**里程碑计划**：
- 模板：标准字段齐全，含公式计算
- 案例：字段一致，增加了SVN地址字段，进度说明更详细

**总体计划**：
- 模板：提供两种样例
- 案例：采用样例1结构（里程碑导向），任务分解更细致

**风险跟踪**：
- 模板：标准字段+指引
- 案例：跟踪日志记录非常详细，形成完整追溯链

### 7.3 最佳实践提炼

1. **模板灵活应用**：根据项目特点调整Sheet结构，但保留核心管控项
2. **里程碑按功能划分**：便于验收和进度监控
3. **风险跟踪日志化**：详细记录跟踪过程，形成闭环
4. **进度说明动态更新**：记录关键节点和变更原因
