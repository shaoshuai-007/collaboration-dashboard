## 二、接口协议设计规范

### 2.1 HTTP接口定义模板

#### 服务定义表格
| 字段 | 说明 |
|------|------|
| 接口描述 | 功能描述 |
| 接口提供方 | 提供方系统名称 |
| 接口调用方 | 调用方系统名称 |
| 参数格式 | application/json |
| 响应格式 | application/json |
| 编码 | UTF-8 |
| 方式或周期 | POST/定时任务 |
| 接口地址 | http://{ip}:{port}/api/xxx |

#### 请求参数表格规范
| 参数名称 | 参数编码 | 类型 | 必填 | 说明 |
|----------|----------|------|------|------|
| 交易流水号 | transactionId | String(100) | 是 | 说明 |

#### 响应参数表格规范
| 参数编码 | 参数类型 | 参数描述 | 是否必须 |
|----------|----------|----------|----------|
| code | String(4) | 状态码说明 | 是 |

#### 响应编码说明表格
| 序号 | 响应编码 | 说明 |
|------|----------|------|
| 1 | 0000 | 操作成功 |
| 2 | 7101 | 具体错误描述 |

### 2.2 接口安全设计

#### 签名机制
```
签名算法：HMAC-Sha256
签名规则：appId=xxx&body=xxx&timestamp=xxx&nonce=xxx（ASCII码升序排列）
密钥：联调时提供
```

#### 加密设计
```
加密算法：AES-256-CBC
```

#### 请求头设计
| 序号 | 字段名 | 是否必填 | 描述 |
|------|--------|----------|------|
| 1 | appId | 是 | 调用方标识 |
| 2 | timestamp | 是 | 时间戳（秒级） |
| 3 | nonce | 是 | 随机字符串（32位） |
| 4 | sign | 是 | 签名结果 |

### 2.3 消息队列接口设计

#### 消息队列申请清单表格
| 序号 | 名称 | 编码 | 生产方 | 消费方 | 用途 |
|------|------|------|--------|--------|------|
| 1 | 商机采集 | mkt_opp_collect | 外围系统 | 中台 | 用途描述 |

#### 消息队列报文格式
- 协议：KAFKA
- 数据格式：JSON
- 参数同HTTP接口

### 2.4 接口设计要点总结

1. **标准化命名**：参数编码使用驼峰命名法（如transactionId、templateCode）
2. **类型标注**：明确类型和长度限制（如String(100)、date）
3. **枚举值文档化**：将枚举值的具体含义在说明中列出
4. **提供报文样例**：包含未加密、加密、成功响应的样例
5. **错误码分类**：按业务场景划分错误码区间

## 三、非功能性设计规范

### 3.1 数据存储设计

#### 大数据量表分析
| 序号 | 表名 | 使用场景 | 数据量预估 |
|------|------|----------|------------|
| 1 | 商机信息 | 查询、单条更新 | 3000万 |
| 2 | 商机属性 | 查询、小批量更新 | 10亿 |

#### 存储方案设计
- 明确存储类型（DORIS/PG）
- 分表方案：按时间或业务维度分表
- 分区方案：按ID取模或hash分区

**示例**：
| 序号 | 表名 | 存储类型 | 分表方案 | 分区方案 |
|------|------|----------|----------|----------|
| 1 | 商机标签 | DORIS | 无 | 无 |
| 2 | 商机信息 | PG | 按月（采集时间） | 商机标识取模，分区数：10 |

#### 数据归档策略
- 数据保留周期：明确年限
- 归档时机：定时任务执行时间
- 归档存储：目标存储位置

### 3.2 性能需求设计

#### 响应时间要求
```
语音转文本：整体识别需在3s内
文本属性提取：整体识别需在5s内
图片属性提取：整体识别需在5s内
```

#### 数据处理要求
- 任务队列并行数配置
- 接口调用失败重试次数（如3次）

### 3.3 安全性需求设计

1. **隐私保护**：属性脱敏规则配置
2. **权限控制**：角色-菜单-功能项-服务权限四级控制
3. **接口认证**：签名机制、加密传输
4. **日志记录**：接口访问日志完整记录
5. **水印保护**：页面水印（姓名+手机号+时间）

### 3.4 可用性设计

1. **定时任务**：扫表策略、队列处理
2. **异常处理**：失败重试机制、异常日志记录
3. **数据一致性**：缓存与数据库同步策略

## 六、技术方案设计规范

### 6.1 功能模块设计模板

```
### 功能描述
简述功能目标和业务价值

### 流程描述
流程图 + 流程步骤说明

### 界面原型
原型图 + 交互说明

### 实现方案
1. 实现逻辑（详细步骤）
2. 模型设计（涉及表结构）
3. 接口设计（调用的服务）

### 物理模型
表结构图示
```

### 6.2 实现方案编写规范

#### 编号分层
```
1. 主步骤
   1.1 子步骤
       1.1.1 具体操作
```

#### 条件分支描述
```
1. 校验规则：必传、是否重复
2. 校验不通过：记录错误信息"xxx"
3. 校验通过：继续下一步
```

#### 异常处理
```
处理过程中调用模型接口失败重试3次
中途处理异常更新状态为"处理异常"，记录失败原因
```

### 6.3 业务规则描述

#### 校验规则示例
```
1. 模板编码校验
   1. 校验规则：模板编码必传、模板在系统中必须存在、模板必须为有效状态
   2. 校验不通过：记录错误信息"采集模板编码不能为空"/"采集模板不存在"/"采集模板已失效"

2. 属性值校验
   1. 校验规则：模板中必填属性是否存在值
   2. 校验不通过：记录错误信息"模板必填属性【{属性名}】属性值为空"
```

#### 数据处理规则示例
```
1. 日期格式修正
   序号 | 非标准格式 | 修正后标准格式
   1    | 20251020   | 2025-10-20
   2    | 2025/10/20 | 2025-10-20
   3    | 2025.10.20 | 2025-10-20
```

## 七、方案设计到详细设计的衔接

### 7.1 方案设计（概要设计）侧重点

| 维度 | 方案设计 | 详细设计 |
|------|----------|----------|
| 目标 | 明确"做什么"、"为什么" | 明确"怎么做" |
| 粒度 | 功能模块级 | 类/方法级 |
| 接口 | 接口协议定义 | 接口详细参数、异常处理 |
| 数据 | 表结构、分区分表 | 字段约束、索引、存储过程 |
| 算法 | 算法策略选择 | 具体实现伪代码 |

### 7.2 详细设计承接内容

从方案设计到详细设计需要展开的内容：

1. **接口设计**
   - 方案设计：接口协议、参数列表
   - 详细设计：参数校验逻辑、异常处理分支、返回值组装逻辑

2. **数据模型**
   - 方案设计：表结构、分区分表策略
   - 详细设计：字段约束规则、索引策略、SQL语句

3. **业务流程**
   - 方案设计：流程步骤、业务规则
   - 详细设计：伪代码、时序图、状态机

4. **非功能性**
   - 方案设计：性能指标、安全策略
   - 详细设计：缓存策略、并发控制、事务管理

### 7.3 详细设计检查清单

- [ ] 每个接口是否有详细的入参校验逻辑？
- [ ] 每个业务流程是否覆盖异常分支？
- [ ] 数据库操作是否有事务边界定义？
- [ ] 是否有缓存使用策略？
- [ ] 是否有日志埋点设计？
- [ ] 是否有监控告警设计？

## 八、学习总结

### 8.1 方案设计文档编写规范

1. **结构清晰**：按照标准目录结构组织内容
2. **图文并茂**：每个设计点配图说明
3. **表格规范**：统一使用标准表格格式
4. **示例完整**：提供报文样例、枚举值说明
5. **版本管理**：明确文档版本和修改记录

### 8.2 技术设计要点

1. **接口设计**：定义清晰、参数规范、安全可靠
2. **数据设计**：考虑数据量、分区分表、归档策略
3. **架构设计**：分层清晰、职责单一、可扩展
4. **非功能性**：性能、安全、可用性、可维护性

### 8.3 编写技巧

1. 使用**表格**展示结构化信息（参数、枚举、字段）
2. 使用**编号**组织层次化步骤
3. 使用**条件分支**描述业务逻辑
4. 使用**示例**说明复杂规则
5. 使用**图示**呈现架构和流程
