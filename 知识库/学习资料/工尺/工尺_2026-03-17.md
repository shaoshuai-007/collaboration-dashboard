## 一、系统设计核心规范

### 1. 接口设计规范（RESTful API）

#### 1.1 基本原则
- **资源导向**: 以资源为核心设计URL，使用名词而非动词
- **HTTP方法语义化**:
  - `GET` - 查询资源（幂等、安全）
  - `POST` - 创建资源（非幂等）
  - `PUT` - 全量更新资源（幂等）
  - `PATCH` - 部分更新资源（幂等）
  - `DELETE` - 删除资源（幂等）
- **状态码规范化**:
  - `2xx` 成功（200/201/204）
  - `4xx` 客户端错误（400/401/403/404/422）
  - `5xx` 服务端错误（500/502/503）

#### 1.2 URL设计规范
```
# 推荐格式
GET    /api/v1/users              # 获取用户列表
GET    /api/v1/users/{id}         # 获取单个用户
POST   /api/v1/users              # 创建用户
PUT    /api/v1/users/{id}         # 更新用户
DELETE /api/v1/users/{id}         # 删除用户

# 关联资源
GET    /api/v1/users/{id}/orders  # 获取用户的订单列表
```

#### 1.3 请求响应规范
```json
// 成功响应
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "timestamp": 1710691200000
}

// 错误响应
{
  "code": 40001,
  "message": "参数校验失败",
  "errors": [
    {"field": "phone", "message": "手机号格式不正确"}
  ],
  "timestamp": 1710691200000
}
```

#### 1.4 分页与过滤
```
GET /api/v1/users?page=1&size=20&sort=created_at:desc
GET /api/v1/users?status=active&role=admin
```

### 2. 数据库设计规范

#### 2.1 表结构设计规范
```sql
-- 基础字段规范
CREATE TABLE users (
    id              BIGINT        PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    user_no         VARCHAR(32)   NOT NULL UNIQUE COMMENT '用户编号',
    phone           VARCHAR(11)   NOT NULL UNIQUE COMMENT '手机号',
    nickname        VARCHAR(64)   DEFAULT NULL COMMENT '昵称',
    status          TINYINT       DEFAULT 1 COMMENT '状态：1正常 2禁用',
    created_at      DATETIME      DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at      DATETIME      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted_at      DATETIME      DEFAULT NULL COMMENT '删除时间（软删除）',
    created_by      VARCHAR(32)   DEFAULT NULL COMMENT '创建人',
    updated_by      VARCHAR(32)   DEFAULT NULL COMMENT '更新人'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

#### 2.2 索引设计规范
| 索引类型 | 使用场景 | 示例 |
|---------|---------|------|
| 主键索引 | 唯一标识 | `PRIMARY KEY (id)` |
| 唯一索引 | 业务唯一约束 | `UNIQUE INDEX uk_phone (phone)` |
| 普通索引 | 高频查询条件 | `INDEX idx_status (status)` |
| 联合索引 | 多条件查询 | `INDEX idx_status_created (status, created_at)` |
| 覆盖索引 | 避免回表 | `INDEX idx_cover (status, name, phone)` |

**索引设计原则**:
- 遵循最左前缀原则
- 区分度高的列放前面
- 避免在低区分度字段建索引（如性别）
- 单表索引数量控制在5个以内

#### 2.3 分表设计规范
```
# 分表策略
1. 垂直分表：按字段拆分（冷热分离）
   - 主表：高频访问字段
   - 扩展表：低频访问字段

2. 水平分表：按数据量拆分
   - 取模分表：user_0 ~ user_99 (适合均匀分布)
   - 范围分表：user_202401 ~ user_202412 (适合时间维度)
   - 一致性哈希：减少扩容迁移量

# 分表键选择
- 选择业务主键（如user_id）
- 保证数据均匀分布
- 避免跨表查询
```

## 二、电信系统设计要点

### 1. AI配案系统接口设计

#### 1.1 业务场景
电信运营商的AI配案系统用于根据用户画像推荐最适合的套餐方案，需要处理：
- 用户数据接入（多源异构数据）
- 实时推荐计算
- 方案匹配与排序
- 结果反馈与优化

#### 1.2 核心接口设计
```
# 接口清单

1. 用户画像查询接口
   POST /api/v1/ai-recommend/profile/query
   Request: { "user_id": "xxx", "phone": "13800138000" }
   Response: { "profile": {...}, "tags": [...] }

2. 套餐推荐接口（核心）
   POST /api/v1/ai-recommend/plan/recommend
   Request: { 
     "user_id": "xxx",
     "scene": "upgrade",  // upgrade/new/renew
     "constraints": { "price_max": 200 }
   }
   Response: { 
     "recommendations": [
       { "plan_id": "P001", "plan_name": "畅享套餐", "score": 0.95, "reason": "..." }
     ]
   }

3. 推荐反馈接口
   POST /api/v1/ai-recommend/feedback
   Request: { "recommend_id": "xxx", "action": "accept", "plan_id": "P001" }

4. 推荐策略配置接口
   POST /api/v1/ai-recommend/strategy/config
   Request: { "scene": "upgrade", "weights": {...}, "rules": [...] }
```

#### 1.3 设计要点
- **响应时间**: 推荐接口 < 500ms（P99）
- **缓存策略**: 用户画像缓存（Redis，TTL=1h）
- **降级方案**: AI服务不可用时返回热门套餐
- **A/B测试**: 支持多策略并行对比

### 3. 高并发系统设计要点

#### 3.1 架构设计原则
```
┌─────────────────────────────────────────────────────────┐
│                      CDN / WAF                          │
├─────────────────────────────────────────────────────────┤
│                    网关层 (API Gateway)                  │
│                 限流 / 认证 / 路由 / 熔断                  │
├──────────────┬──────────────┬───────────────────────────┤
│   服务集群A   │   服务集群B   │       服务集群N           │
│  (无状态)     │  (无状态)     │       (无状态)            │
├──────────────┴──────────────┴───────────────────────────┤
│                      消息队列                            │
│              (异步解耦 / 削峰填谷)                        │
├─────────────────────────────────────────────────────────┤
│              缓存层 (Redis Cluster)                      │
│          热点数据 / 分布式锁 / 计数器                      │
├─────────────────────────────────────────────────────────┤
│              数据库层 (MySQL Cluster)                    │
│          读写分离 / 分库分表 / 主从复制                    │
└─────────────────────────────────────────────────────────┘
```

#### 3.2 核心技术要点

| 场景 | 技术方案 | 说明 |
|-----|---------|------|
| 高并发读 | 多级缓存 | CDN → 本地缓存 → Redis → DB |
| 高并发写 | 异步写入 | MQ削峰 + 批量写入 |
| 热点数据 | 本地缓存 + Redis | 防止缓存穿透、击穿、雪崩 |
| 分布式事务 | 最终一致性 | TCC / Saga / 本地消息表 |
| 限流熔断 | 多级防护 | 网关限流 → 服务限流 → DB限流 |
| 服务降级 | 有损服务 | 核心业务优先，非核心可降级 |

#### 3.3 电信场景特殊考量
```
1. 实时性要求
   - 话单处理：延迟 < 100ms
   - 推荐计算：响应 < 500ms
   - 方案：流式计算 + 预计算

2. 数据一致性
   - 余额扣减：强一致性
   - 推荐结果：最终一致性可接受
   - 方案：分布式事务 + 对账机制

3. 容灾要求
   - RTO < 5分钟
   - RPO = 0（核心数据）
   - 方案：多活架构 + 异地容灾

4. 数据安全
   - 手机号脱敏
   - 通话记录加密存储
   - 访问日志审计
```
