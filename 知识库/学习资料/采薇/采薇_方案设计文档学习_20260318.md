## 二、需求到方案的衔接方法

### 2.1 建设背景与目的的描述方式

**文档2范例（需求方案）**：
```
建设背景：
- 一线诉求问题：隐私顾虑、场景化采集配置、空间数据标准化、部分角色工具缺失

建设目的：
- 解决一线诉求问题
- 建设标准化商机笔记本嵌入一线工具，实现商机数字化采集
```

**文档1范例（概要设计）**：
```
业务目标：
通过"统一标准、智能工具、闭环管控、数据赋能"，打造数字化运营新空间，
实现商机管理从"线下分散"到"线上集中"、从"人工低效"到"智能高效"的转型，
最终提升商机转化率、优化资源配置、增强市场竞争力。
```

**衔接要点**：
1. 需求文档侧重**问题导向**：描述现状问题、用户痛点
2. 设计文档侧重**目标导向**：描述业务价值、转型方向、量化目标
3. 衔接方式：问题→解决方案→业务目标

---

### 2.2 业务流程描述方式

**需求方案中的流程描述**：
- 使用业务流程图（如有）
- 文字描述操作步骤
- 界面原型配合说明

**概要设计中的流程描述**：
- **流程图**：明确数据流转方向
- **实现逻辑**：伪代码形式的处理步骤
- **状态机**：状态流转规则

**范例（商机采集流程）**：
```
实现逻辑：
1. 定时任务扫描到达日志状态为"待处理"
2. 更新日志状态为"处理中"
3. 查询采集属性转换配置
4. 分页查询目标表数据
5. 逐条转换和推送
6. 更新推送状态
7. 处理完成更新日志状态
```

**衔接要点**：
- 需求描述"做什么"：用户视角的业务操作
- 设计描述"怎么做"：技术实现的处理流程
- 流程衔接需要明确：触发条件、处理步骤、输出结果

---

### 2.3 功能清单编写规范

**需求方案的功能清单结构**：
| 字段 | 说明 |
|------|------|
| 功能描述 | 一句话概括功能目的 |
| 业务流程 | 用户操作步骤 |
| 界面原型 | 可视化展示 |
| 详细说明 | 字段规则、交互规则、校验规则 |

**概要设计的功能描述结构**：
| 章节 | 内容 |
|------|------|
| 功能描述 | 功能概述 |
| 流程描述 | 技术流程图 |
| 界面原型 | 可选 |
| 实现方案 | 详细技术实现逻辑 |
| 物理模型 | 数据表设计 |

**功能清单衔接要点**：
1. **需求侧**：关注用户能做什么、怎么操作
2. **设计侧**：关注如何实现、数据如何流转
3. **衔接桥梁**：
   - 需求中的"字段规则" → 设计中的"数据模型"
   - 需求中的"操作流程" → 设计中的"接口调用链路"
   - 需求中的"校验规则" → 设计中的"业务规则实现"

### 2.4 用户角色和权限设计

**需求方案中的角色定义**：
```
角色权限表：
| 分类 | 角色 | 功能权限 | 数据权限 |
|电信员工|店长|登录、个人信息、在线录商机...|查看所在厅店员工商机数据（脱敏）|
|电信员工|网格经理|...|查看所在网格员工商机数据（脱敏）|
|非电信人员|其他|登录、注册、我的商机|查看我录入的商机|
```

**概要设计中的权限设计**：
```
权限控制：
1. 菜单权限：各角色具备的菜单访问权限
2. 数据权限：根据用户岗位角色的data_privilege确认数据权限层级
3. 角色权限：根据orgId结合data_privilege确认数据权限范围
4. 脱敏规则：根据角色商机属性脱敏规则展示数据
```

**权限设计衔接要点**：
1. **需求定义角色**：明确有哪些角色、每个角色能做什么
2. **设计定义权限体系**：
   - 功能权限（菜单、按钮、服务）
   - 数据权限（组织架构层级）
   - 字段权限（脱敏规则）
3. **权限配置化**：设计文档中体现权限可配置的能力

### 2.5 数据模型设计

**需求方案中的数据准备**：
- 元素清单
- 预设模板
- 脱敏字段说明
- 打标数据规则

**概要设计中的数据模型**：
```
目标表约束：
- 源系统商机唯一标识（必填）
- 商机归属类型（选填）
- 采集时间（必填）
- ...

存储方案：
- 分区分表策略
- 数据归档规则
```

**数据模型衔接要点**：
1. 需求文档定义**业务字段**：用户需要录入/查看的字段
2. 设计文档定义**物理模型**：表结构、索引、分区策略
3. 衔接需要明确：
   - 字段映射关系
   - 枚举值定义
   - 数据字典

## 三、需求到方案的转化技巧

### 3.1 需求分类转化

| 需求类型 | 需求文档呈现 | 设计文档呈现 |
|----------|-------------|-------------|
| 功能需求 | 操作流程、界面原型 | 实现方案、接口设计 |
| 非功能需求 | 性能要求、安全要求 | 架构设计、技术选型 |
| 数据需求 | 字段定义、数据规则 | 数据模型、存储方案 |
| 接口需求 | 对接系统、数据格式 | 接口协议、消息队列 |

### 3.2 关键衔接元素

1. **功能点映射**：每个需求功能点对应设计章节
2. **数据流映射**：业务数据流对应技术数据流
3. **接口映射**：交互操作对应API接口
4. **状态映射**：业务状态对应数据状态字段

### 3.3 描述层次差异

| 层面 | 需求文档 | 设计文档 |
|------|----------|----------|
| 视角 | 用户视角 | 开发视角 |
| 语言 | 业务术语 | 技术术语 |
| 细节 | 操作细节 | 实现细节 |
| 粒度 | 功能模块 | 技术模块 |

## 四、优秀实践总结

### 4.1 文档结构规范

**需求方案标准结构**：
1. 简介（背景、目的）
2. 总体概述
3. 功能模块详细说明
4. 数据说明
5. 非功能性需求
6. 待确定问题

**概要设计标准结构**：
1. 文档说明
2. 需求描述
3. 总体方案（业务目标、方案概述、架构图、角色权限）
4. 功能详细设计
5. 接口协议
6. 非功能性设计
7. 附录

### 4.2 优秀实践

1. **版本控制**：建立/修改记录表
2. **术语定义**：统一业务术语
3. **界面原型**：需求必配，设计可选
4. **流程图**：设计文档必须有
5. **表格规范**：字段定义、枚举值用表格呈现
6. **待定问题**：记录未明确事项

### 4.3 衔接检查清单

- [ ] 需求功能点是否全部覆盖设计
- [ ] 数据字段是否完整映射
- [ ] 业务流程是否有对应技术流程
- [ ] 权限规则是否完整定义
- [ ] 接口需求是否明确
- [ ] 非功能需求是否考虑

## 五、个人收获与反思

### 5.1 核心收获

1. **需求分析是桥梁**：需求分析师的核心价值在于将业务需求转化为可落地的技术方案
2. **结构化思维**：从问题→目标→方案→实现的逻辑链条
3. **细节决定质量**：字段定义、枚举值、校验规则等细节影响方案完整性
4. **文档即沟通**：清晰的文档结构降低开发理解成本

### 5.2 能力提升方向

1. 学习绘制系统架构图、数据流图
2. 熟悉常用的数据模型设计方法
3. 掌握业务流程到技术流程的转化技巧
4. 积累行业领域知识（电信行业）

### 5.3 后续行动

1. 尝试按照学习到的结构编写需求文档
2. 对照范本文档自查现有工作产出
