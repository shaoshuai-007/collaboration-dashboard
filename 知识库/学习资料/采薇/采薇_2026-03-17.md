## 一、需求分析方法论

### 1.1 用户故事编写规范

**标准格式**: As a... I want... So that...

```
As a [用户角色]
I want [功能需求]
So that [业务价值/收益]
```

**示例**:
- As a 电信客户经理，I want 一键生成客户套餐推荐方案，So that 提升服务效率，缩短客户等待时间。

**编写要点**:
| 要素 | 说明 | 常见问题 |
|------|------|----------|
| 角色定义 | 清晰明确，避免模糊 | "用户"太泛，需具体化 |
| 需求描述 | 简洁可测试 | 避免技术术语 |
| 价值说明 | 体现业务意义 | 避免"因为需要" |

**INVEST原则验证**:
- **I**ndependent - 独立性
- **N**egotiable - 可协商
- **V**aluable - 有价值
- **E**stimable - 可估算
- **S**mall - 小而精
- **T**estable - 可测试

---

### 1.2 需求优先级评估方法（MoSCoW法则）

| 优先级 | 英文含义 | 中文含义 | 说明 |
|--------|----------|----------|------|
| **M** | Must have | 必须有 | 核心功能，没有则项目失败 |
| **S** | Should have | 应该有 | 重要但可延后或替代 |
| **C** | Could have | 可以有 | 锦上添花，时间允许再做 |
| **W** | Won't have | 暂不做 | 本次迭代排除，记录备选 |

**评估维度**:
1. 业务价值（对客户/企业的价值）
2. 实施成本（时间、人力、资源）
3. 风险程度（技术风险、合规风险）
4. 依赖关系（前置条件、系统集成）

**电信项目优先级示例**:
```
M - 用户身份认证与鉴权
S - 智能套餐推荐引擎
C - 客户行为分析可视化大屏
W - AI智能客服（二期规划）
```

---

### 1.3 需求验收标准制定（SMART原则）

| 要素 | 英文 | 含义 | 验收标准示例 |
|------|------|------|--------------|
| **S** | Specific | 具体明确 | "系统响应时间<3秒" |
| **M** | Measurable | 可量化 | "支持1000并发用户" |
| **A** | Achievable | 可实现 | 技术可行、资源可控 |
| **R** | Relevant | 相关性 | 与业务目标一致 |
| **T** | Time-bound | 有时限 | "2026年Q2上线" |

**验收标准模板**:
```markdown
### 需求ID: REQ-001
**需求名称**: 套餐推荐功能

**验收标准**:
- [ ] Given 用户登录系统，When 访问推荐页面，Then 显示3-5个推荐套餐
- [ ] 推荐准确率 >= 85%
- [ ] 响应时间 <= 2秒（P95）
- [ ] 支持PC端和移动端访问
```

## 二、电信业务需求特点

### 2.1 AI智能配案系统需求分析要点

**核心业务场景**: 根据客户画像、消费习惯、历史行为，智能匹配最优套餐方案。

**需求分析框架**:

```
┌─────────────────────────────────────────────────────┐
│                 AI智能配案系统                        │
├─────────────────────────────────────────────────────┤
│  输入层          │  处理层           │  输出层       │
├─────────────────┼───────────────────┼───────────────┤
│ • 客户基本信息    │ • 数据清洗引擎     │ • 推荐套餐    │
│ • 消费行为数据    │ • 特征提取模块     │ • 配案理由    │
│ • 历史订购记录    │ • AI推理模型       │ • 优惠策略    │
│ • 竞品分析数据    │ • 规则引擎         │ • 预期收益    │
└─────────────────────────────────────────────────────┘
```

**关键需求点**:

| 需求类别 | 具体需求 | 验收标准 |
|----------|----------|----------|
| 数据接入 | 多源数据整合 | 支持5+数据源实时接入 |
| 模型能力 | 推荐准确率 | ≥85% Top-5准确率 |
| 响应性能 | 实时推荐 | ≤3秒返回结果 |
| 可解释性 | 推荐理由 | 每个推荐附带3条理由 |
| 合规性 | 数据脱敏 | 敏感信息100%脱敏 |

**技术约束**:
- 需对接现有CRM系统
- 模型需支持灰度发布
- 日志审计要求满足《网络安全法》

### 2.2 数据安全项目需求拆解方法

**需求拆解四步法**:

```
Step 1: 合规对标 → 识别适用的法律法规
Step 2: 风险评估 → 数据资产盘点 + 威胁分析
Step 3: 场景映射 → 业务场景与数据流梳理
Step 4: 需求细化 → 功能需求 + 非功能需求
```

**数据安全需求分类**:

| 类别 | 需求项 | 典型措施 |
|------|--------|----------|
| **数据采集** | 最小必要原则 | 数据采集声明、授权确认 |
| **数据存储** | 分类分级存储 | 加密存储、访问控制 |
| **数据传输** | 加密传输 | TLS/SSL、VPN通道 |
| **数据使用** | 权限管控 | 最小权限、操作审计 |
| **数据销毁** | 安全删除 | 物理销毁、逻辑删除 |

**需求拆解示例**:

```
【原始需求】保护客户隐私数据

【拆解后】
├── M-001: 数据分类分级管理
│   ├── M-001-1: 建立数据分类标准（个人识别信息、消费记录等）
│   └── M-001-2: 数据打标工具，支持自动分类
├── M-002: 敏感数据脱敏
│   ├── M-002-1: 静态脱敏（开发测试环境）
│   └── M-002-2: 动态脱敏（生产查询场景）
└── S-001: 数据访问审计
    ├── S-001-1: 访问日志记录（谁、何时、访问什么）
    └── S-001-2: 异常访问告警
```

## 三、核心方法论总结

### 方法论速查表

| 方法论 | 应用场景 | 核心价值 | 关键输出 |
|--------|----------|----------|----------|
| 用户故事 | 需求捕获 | 从用户视角描述需求 | As a...I want...So that... |
| MoSCoW | 需求优先级 | 资源约束下的取舍决策 | M/S/C/W分类列表 |
| SMART | 验收标准 | 需求可测试、可验证 | 具体可量化的验收条件 |

### 方法论组合应用流程

```
需求收集 → 用户故事编写 → MoSCoW优先级排序 → SMART验收标准制定 → 需求评审
```

## 四、电信业务需求分析要点

### 4.1 行业特点

1. **合规要求高**: 个人信息保护法、网络安全法、行业监管规定
2. **系统复杂度高**: 多系统对接、遗留系统集成
3. **数据规模大**: 亿级用户、海量实时数据
4. **安全性要求严**: 数据脱敏、加密、审计全覆盖

### 4.2 需求分析注意事项

| 注意点 | 说明 | 应对策略 |
|--------|------|----------|
| 法规遵循 | 个人信息保护、数据安全 | 需求清单对标法规条文 |
| 系统边界 | 与现有系统集成 | 明确接口定义和数据流向 |
| 性能要求 | 高并发、低延迟 | 提前进行容量规划 |
| 可扩展性 | 业务快速迭代 | 模块化、松耦合设计 |

### 4.3 需求文档关键章节

```markdown
1. 项目背景与目标
2. 业务需求描述（用户故事）
3. 功能需求清单（MoSCoW分级）
4. 非功能需求（性能、安全、可用性）
5. 验收标准（SMART原则）
6. 系统接口需求
7. 数据需求（数据字典、数据流）
8. 合规与安全需求
```

## 五、学习心得

作为需求分析师，需要：

1. **方法论是工具，不是教条** - 灵活运用，根据项目特点调整
2. **电信行业合规是底线** - 需求分析前先吃透法规要求
3. **用户故事要讲好故事** - 角色要具体，价值要明确
4. **优先级是决策工具** - 资源有限时帮项目做减法
5. **验收标准是质量门** - 写得越具体，后期扯皮越少
