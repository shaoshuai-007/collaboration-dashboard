## 二、核心编写要点

### 2.1 简介部分

| 章节 | 编写要点 |
|------|---------|
| 建设背景 | 一段话描述业务背景，涵盖客户原始要求、集团派发单要求 |
| 建设目的 | 两个层面：①解决问题（简述要解决什么）②效果预估（量化目标） |
| 引用资料 | 完整列出引用的文档，可删减 |
| 名词解释 | 列表形式，术语名称+解释+备注，可删减 |

### 2.2 业务需求

**业务场景 → 5W1H分析法**：
| 要素 | 含义 | 示例方向 |
|------|------|---------|
| Who | 用户角色 | 谁使用这个功能 |
| When | 触发时机 | 什么场景触发 |
| Where | 使用场景 | 在什么地方使用 |
| What | 业务内容 | 具体做什么事 |
| Why | 业务价值 | 为什么要做 |
| How | 实现方式 | 怎么实现 |

**业务流程**：
- 从业务角度描述场景流程
- 侧重点：不同角色的操作场景
- 必须：与业务方确认流程

**现状分析**（改造需求必选）：
- 业务现状：当前界面、流程、逻辑
- 目前问题：可用对比图呈现"现状vs目标"

### 2.3 功能需求

**功能清单表格字段**：
- 功能模块
- 功能模块概述
- 使用角色
- 改造（新建or改造）
- 优先级

**单个功能需求包含6个板块**：
1. 功能描述
2. 流程描述
3. 交互原型图
4. 业务描述
5. 数据描述
6. 影响范围

## 三、流程图类型选择（重点）

| 图类型 | 适用场景 |
|-------|---------|
| 流程图 | 系统内的业务规则判定、处理分支逻辑 |
| 泳道图 | 系统之间有交互，重点突出业务判定逻辑 |
| 时序图 | 系统之间交互处理环节多，重点突出交互逻辑 |
| 数据逻辑图 | 重点突出不同环节的数据处理和保存差异 |
| 状态变更流转图 | 多个状态变更的处理逻辑 |

**选择原则**：
- 单系统内部逻辑 → 流程图
- 多系统交互+业务判定 → 泳道图
- 多系统交互+交互逻辑 → 时序图
- 数据处理差异 → 数据逻辑图
- 状态流转 → 状态变更流转图

## 四、数据描述规范

**数据模型设计包含**：
- 数据涉及的表、字段
- 数据处理规则
- 数据权限说明（查看、操作权限）
- 数据模型设计（新建表、修改表字段）

**表结构文档格式**：
```
表名 `table_name` (新建/修改，分表规则)

| 字段 | 名称 | 数据类型 | 备注 |
|------|------|---------|------|
| field_name | 字段名称 | VARCHAR(50) | 说明 |
```

## 五、接口需求规范

**接口文档结构**：
1. 接口清单（所有接口汇总）
2. 单个接口详细描述：
   - 接口描述（用来干什么）
   - 接口方式（数据库同义词/直接访问/协议等）
   - 接口参数（出入参、数据封装方式和类型、参考案例）
3. 外围配合需求

## 六、非功能需求

### 6.1 安全性需求
- 业务安全（F12调试风险）
- 数据安全（加密脱敏）
- 访问安全（注入、鉴权）

### 6.2 性能需求（量化指标）

**系统整体性能**：
| 指标 | 要求 |
|------|------|
| 在线用户数 | 如200w |
| 平稳运行时间 | 如7×24h |
| 平均响应时间 | 如页面<2s |
| CPU使用率 | <75% |

**不同功能性能需求**：
- 高频功能：系统高频使用
- 关键功能：登录、注册、支付等不能出问题
- 特色功能：产品卖点
- 大数据量功能：报表查询

### 6.3 其他非功能需求
- 网络需求：内网/外网/VPN/特殊网段
- 兼容性需求：浏览器、手机机型
- 埋点需求：菜单模块、页面、埋点位置、记录数据
- 体验需求：交互点要求、评审问题记录

## 八、核心原则记忆口诀

> **完清可溯可验证**

## 九、常见问题避坑

| 问题 | 解决方案 |
|------|---------|
| 需求描述模糊 | 使用5W1H法分析 |
| 流程不完整 | 补充异常流程和异常处理 |
| 接口定义不清 | 明确出入参、数据格式 |
| 性能要求不具体 | 量化指标（响应时间、TPS等） |
