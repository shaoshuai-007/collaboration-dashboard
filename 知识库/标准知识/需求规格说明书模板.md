## 三、业务需求

### 3.1 业务场景
> 从业务角度出发，要实现的业务/用户场景（非系统功能），等同于用户故事

**分析方法：5W1H法**
- Who（谁）：用户角色
- When（什么时间）：触发时机
- Where（什么地点）：使用场景
- What（什么事情）：业务内容
- Why（为什么）：业务价值
- How（如何达成）：实现方式

### 3.2 业务流程
> 从业务角度出发，描述要实现的业务场景流程
> - 侧重点：不同角色的操作场景
> - 与业务方确认流程

### 3.3 现状分析（改造需求必选）
> - 业务现状：目前的界面、业务流程、逻辑等
> - 目前问题：可用对比图呈现"现状vs目标"

**流程图类型选择**：
| 图类型 | 适用场景 |
|-------|---------|
| 流程图 | 系统内的业务规则判定、处理分支逻辑 |
| 泳道图 | 系统之间有交互，重点突出业务判定逻辑 |
| 时序图 | 系统之间交互处理环节多，重点突出交互逻辑 |
| 数据逻辑图 | 重点突出不同环节的数据处理和保存差异 |
| 状态变更流转图 | 多个状态变更的处理逻辑 |

#### 4.3.5 数据描述

## 六、非功能需求

### 6.1 安全性需求
> 需要考虑的安全策略：
> - 业务安全（F12）
> - 数据安全（加密脱敏等）
> - 访问安全（注入、鉴权）

### 6.2 性能需求

**系统整体性能需求**：
| 指标 | 要求 |
|------|------|
| 在线用户数 | 如支持在线用户数200w |
| 平稳运行时间 | 如7×24h |
| 平均响应时间 | 如页面打开时间低于2s |
| CPU使用率 | <75% |

**不同功能/接口性能需求**：
- 高频功能：系统中高频率使用的功能
- 关键功能：系统中不能出现问题的功能（登录、注册、支付）
- 特色功能：系统中的亮点功能，产品的卖点
- 大数据量功能：报表查询

**示例**：登录功能
- 并发用户数：500
- 响应时间：2s
- TPS：500/s
- CPU：不得超过75%

## 八、编写要点总结

### 8.1 核心原则

1. **完整性**：覆盖所有功能点和业务场景
2. **清晰性**：描述准确，无歧义
3. **可追溯性**：引用资料、版本记录完整
4. **可验证性**：业务规则、验收标准明确

### 8.2 常见问题

| 问题 | 解决方案 |
|------|---------|
| 需求描述模糊 | 使用5W1H法分析 |
| 流程不完整 | 补充异常流程和异常处理 |
| 接口定义不清 | 明确出入参、数据格式 |
| 性能要求不具体 | 量化指标（响应时间、TPS等） |
